# Анализ уточнений к плану

## 1. Field-level LWW — ⚠️ Нужно уточнить: а оно того стоит для MVP?

**Идея правильная**, но давай разберём цену:

**Что придётся сделать:**
- Для каждой сущности вместо одного `updated_at` хранить `field_timestamps` — JSON-объект вида:
```json
{
  "title": "2026-04-06T14:00:00Z",
  "priority": "2026-04-06T14:05:00Z",
  "description": "2026-04-06T13:50:00Z"
}
```
- На сервере при merge — сравнивать **каждое поле** по отдельности
- В IndexedDB — хранить эти метки тоже
- При каждом обновлении сущности — обновлять timestamp только изменённых полей
- Sync payload увеличивается (передаём не просто `updated_at`, а словарь)

**Взвешенная оценка для однопользовательского приложения:**

Конфликт "изменил title на устройстве A, изменил priority на устройстве B" — **реально ли он частый у одного пользователя?** Для этого нужно быть офлайн на обоих устройствах и редактировать одну и ту же задачу. Это крайне редкий сценарий.

**Рекомендация:** Для MVP оставить entity-level LWW, но **не выбрасывать проигравшую версию молча**, а логировать её (см. пункт 8). Field-level LWW — в следующую итерацию, если реально столкнёшься с потерей данных. Иначе это оверинжиниринг для MVP однопользовательского приложения.

## 2. Delete vs Update конфликт — ✅ Хорошее уточнение, но нужно продумать реализацию

**Согласен, это правильное изменение.** Но нужно уточнить:

- **Где показывать конфликт?** У тебя UI для конфликтов — Phase 5+. Значит на Phase 4 нужно хотя бы:
  - Soft-delete оставить как есть (сущность помечена `is_deleted = true`)
  - Но **не применять delete локально** на устройстве, где были правки
  - Сохранить конфликт в таблицу `conflicts` в IndexedDB
  - На Phase 5 показать: "Удалено на другом устройстве. Восстановить?"

- **Что делать, если пользователь игнорирует?** Нужен дефолтный таймаут или поведение: "если конфликт не разрешён за 30 дней — удаление побеждает"

- **Что с каскадами?** Если удалён project, а на другом устройстве добавлены tasks в этот project — это тоже конфликт? Каждый task будет отдельным конфликтом? Это может быть шумно.

**Рекомендация:** Реализовать, но **только для прямых удалений** (user явно удалил). Каскадные удаления зависимых сущностей — пусть пока применяются автоматически с логированием.

## 3. Нормализация времени по серверу — ✅ Принять

**Это самое простое и правильное решение для твоего случая.** Уточнений почти не нужно, но опиши в плане:

```
1. Каждый ответ /sync возвращает sync_timestamp (серверное время)
2. Клиент вычисляет clock_offset = server_time - local_time
3. Сохраняет clock_offset в syncMeta
4. При установке updated_at: Date.now() + clock_offset
5. Пересчитывать offset при КАЖДОМ успешном sync (а не один раз)
```

**Единственный нюанс:** при **самом первом** запуске (до первого синка) offset = 0. Это нормально — первый синк всё выровняет.

## 4. Транзакционность обработки операций — ✅ Нужно уточнить и зафиксировать

Текущий план описывает dependency-ordered processing (areas → contexts → ... → subtasks), но не говорит, что делать при ошибке.

**Мой совет — partial-apply с возвратом ошибок**, потому что:

- "Всё или ничего" означает, что одна битая операция (например, невалидный project) заблокирует синк **всех** 50 операций в батче, включая независимые (areas, tags)
- Для однопользовательского приложения это значит: одна ошибка = застрял синк = копятся pending_ops

**Зафиксировать в плане:**

```
Стратегия: partial-apply

1. Обрабатываем операции в dependency order
2. Если операция X упала:
   - Логируем ошибку
   - Пропускаем все операции, ЗАВИСЯЩИЕ от X
   - Продолжаем с независимыми
3. Ответ содержит:
   - applied: [...успешные...]
   - errors: [{ op_id, error_code, message, dependent_skipped: [op_ids] }]
4. Клиент:
   - Помечает applied ops как synced
   - Ошибочные ops остаются в pending (retry при следующем sync)
   - После 5 неудачных попыток одной op → показать ошибку пользователю
```

## 5. Sync triggers: polling + visibilitychange + Web Push — ✅ Правильный набор

Для твоего стека (Android + Windows + Linux) — отличная комбинация:

| Trigger | Android (Chrome) | Windows (Chrome/Edge) | Linux (Chrome/Firefox) |
|---------|------|---------|-------|
| `online` event | ✅ | ✅ | ✅ |
| `visibilitychange` | ✅ | ✅ | ✅ |
| Periodic 5 мин | ✅ | ✅ | ✅ |
| Manual button | ✅ | ✅ | ✅ |
| Web Push API | ✅ | ✅ | ✅ |

**Web Push работает на всех трёх!** И в отличие от SSE, он работает когда приложение закрыто.

**Уточнения для плана:**
- Web Push — Phase 5 (вместо SSE)
- Нужен VAPID key pair на сервере
- Нужна подписка пользователя на push (PushManager.subscribe)
- Push notification → показать notification "Данные обновлены" → при клике открыть приложение → sync
- Или silent push (без показа notification) — просто триггерит sync в Service Worker. **Но!** Некоторые браузеры требуют показывать notification при push. Проверить для каждого.

**Убрать из плана:** SSE (заменяется Web Push для твоего стека).

## 6. Background Sync API — ⚠️ Решение: оставить, но как progressive enhancement

Раз ты на Android + Windows + Linux (всё Chrome/Edge), Background Sync **реально работает** на всех твоих устройствах! Это не маргинальное улучшение для тебя.

**Что он даёт:** если пользователь внёс изменения и закрыл вкладку, Chrome сам вызовет `sync` event в Service Worker когда появится сеть. Без этого — изменения будут ждать в pending_ops до следующего открытия приложения.

**Рекомендация:** 

```
- Оставить Background Sync
- Но реализовать ПОСЛЕДНИМ среди sync triggers
- Код буквально 15 строк:

// При добавлении pending_op:
if ('serviceWorker' in navigator && 'SyncManager' in window) {
  const reg = await navigator.serviceWorker.ready;
  await reg.sync.register('sync-pending-ops');
}

// В service-worker.js:
self.addEventListener('sync', event => {
  if (event.tag === 'sync-pending-ops') {
    event.waitUntil(doSync());
  }
});
```

Это **не** сложность ради маргинального улучшения — для твоего стека это 15 строк с реальной пользой.

## 7. In-memory флаг + navigator.locks — ✅ Принять, но с оговоркой

**Согласен, `recoverStaleSyncFlag` — это костыль.** `navigator.locks` — правильное решение.

```javascript
async function doSync() {
  await navigator.locks.request('sync-lock', { ifAvailable: true }, async (lock) => {
    if (!lock) return; // другая вкладка уже синкает
    await actualSync();
  });
}
```

**Оговорка:** `navigator.locks` поддерживается в Chrome 69+, Firefox 96+, Safari 15.4+. Для твоего стека (Android/Windows/Linux) — **всё ок**.

**Но** — нужен fallback для случая, когда sync вызывается из Service Worker (Background Sync). Service Worker не имеет доступа к тем же locks, что и страница. Уточнить: у `navigator.locks` в SW — [он работает](https://developer.mozilla.org/en-US/docs/Web/API/LockManager), locks шарятся между page и SW в рамках одного origin. Значит **всё ок, работает**.

**Удалить из плана:** весь `recoverStaleSyncFlag`, `sync_in_progress` в IndexedDB, `sync_started_at`.

## 8. Убрать conflicts из API на Phase 4 — ⚠️ Частично

**Согласен, что полноценный conflicts array не нужен на Phase 4.** Но я бы не убирал его полностью, а **упростил**:

```
Phase 4 ответ /sync:
{
  sync_timestamp: "...",
  applied: [{ client_id, server_id, op_id, status: "applied" | "lww_overwritten" }],
  server_changes: [...]
}
```

Добавить поле `status: "lww_overwritten"` к applied ops, чтобы клиент знал: "твоя операция была применена, но потом перезаписана более новой версией". Это стоит 1 строчку кода на сервере, но даёт:
- Возможность логировать на клиенте "что было перезаписано"
- Готовность к Phase 5 (просто начать показывать эти записи в UI)

**Убрать:** полный `conflicts` array с `server_data`, `local_data`, `reason`.
