## Step 2 — Assumption Inversion

### Инверсия 1: Field-merge по полям

```
Assumption: Для каждой задачи нужно отслеживать изменения по отдельным полям 
            (title, description, gtd_status и т.д.) с отдельными timestamps
Inversion:  Бекенд не возвращает updated_at по полям — только один updated_at на запись.
            Отслеживать field-level timestamps придётся только на клиенте.
Impact:     При pull с сервера невозможно узнать, какое конкретно поле изменилось.
            «Поле-за-полем» превращается в LWW на уровне записи, потому что сервер 
            не даёт гранулярности.
Mitigation: (а) Упростить до LWW на уровне записи — сравниваем task.updated_at.
            (б) Если нужен field-merge — добавить на бекенде поле-level timestamps 
                или хранить историю изменений на сервере. Это отдельная задача на бекенд.
```

### Инверсия 2: Dexie полностью заменит React Query

```
Assumption: useLiveQuery из dexie-react-hooks покроет все кейсы React Query 
            (реактивные подписки, кэширование, инвалидация, optimistic updates)
Inversion:  useLiveQuery — это только реактивная подписка на IDB-запрос.
            Нет: stale/gc timers, retry, prefetch, suspense, parallel queries,
            automatic background refetch, devtools.
Impact:     Потеря удобных фич React Query: нет Devtools для инспекции данных,
            нет auto-refetch по таймеру, нет retry при ошибках чтения из IDB,
            нет suspense-паттерна. Каждую фичу придётся reimplement.
Mitigation: Оценить стоимость потери каждой RQ-фичи. Для большинства фич 
            (staleTime, retry, prefetch) — они не нужны при local-first, 
            потому что данные уже локальные. Но Devtools и error boundaries — 
            нужно продумать отдельно.
```

### Инверсия 3: Синхронизация проста — pull/push

```
Assumption: SyncEngine — это простой pull (GET /tasks) + push (POST/PUT queued mutations).
            Сервер — source of truth, клиент — replica.
Inversion:  Синхронизация — сложная задача. Нужно обрабатывать:
            - Порядок мутаций (create → update → delete — зависят от порядка)
            - Создание офлайн (нет серверного ID) → при sync сервер назначает ID
            - Удаление офлайн → при sync задача уже может быть изменена на сервере
            - Paging при pull (10к задач)
            - Partial failure (5 из 10 мутаций успешно, 5 нет)
            - Дублирование (мутация выполнена, но ack не дошёл)
Impact:     SyncEngine — самый сложный компонент. Оценка «реализовать SyncEngine» 
            в рамках шага 1.3 наряду с переписыванием хуков — нереалистична.
Mitigation: Вынести SyncEngine в отдельную итерацию. Сначала заставить работать 
            локальный CRUD без синхронизации (офлайн-only), потом добавить sync.
```

## Step 3 — Missing Scenarios

| Сценарий | Риск | Обработка |
|----------|------|-----------|
| **Пользователь создал задачу офлайн, затем удалил до синхронизации** | 🟡 | Нужно отменить queued create, не отправлять на сервер. План не описывает отмену мутаций |
| **Пользователь на двух вкладках одновременно** | 🟡 | Dexie работает в одном origin, но две вкладки = два экземпляра Dexie. Нужна обработка `versionchange` / `blocked` событий |
| **IndexedDB quota exceeded** (браузер ограничил ~50MB) | 🟡 | Нет обработки. При 10к задач с описаниями — возможно. Нужно добавить cleanup стратегию |
| **Пользователь вышел из аккаунта** — что происходит с локальными данными? | 🔴 | Данные чужого пользователя останутся в Dexie. Нужен очистка при logout |
| **Бекенд изменил схему (миграция)** — клиентский Dexie-схема не совпадает | 🟡 | Нужна стратегия версионирования Dexie-схемы + принудительный re-sync |
| **Пользователь открыл приложение в приватном режиме** — IndexedDB удаляется при закрытии | 🟢 | Все данные потеряются. Acceptable? Документировать |
| **Pull-синхронизация: сервер вернул данные, которых нет локально (другой пользователь? баг?)** | 🟡 | Нужна валидация server response перед записью в Dexie |
| **Множество rapid-мутаций одной задачи** (быстро toggle 5 раз) | 🟢 | localChanges перезапишется. Для toggle — только финальное состояние важно |

### Рекомендации по доработке плана

1. **Заменить field-merge на LWW** — бекенд не поддерживает field-level timestamps. LWW на уровне записи (по `updated_at`) решает 99% кейсов для одноместного приложения.

2. **Определить Dexie-схему** — таблицы, поля, индексы, типы. Это фундамент всего плана.

3. **Описать SyncEngine протокол** — эндпоинты, формат запросов/ответов, стратегия initial sync, обработка ошибок.

4. **Определить стратегию ID** — клиентский UUID для новых задач, как сервер их принимает.

5. **Разбить итерацию 1 на 1a (инфраструктура) и 1b (хуки)** — сначала Dexie + SyncEngine как библиотека, потом переписывание хуков.
