## Постановка проблемы

Приложение должно полноценно работать при отсутствии подключения к бекенду. Текущее состояние:
- Редактирование задачи (PUT) — работает офлайн (благодаря `updateTaskMutation.onMutate` + `localTaskChanges.ts` + IndexedDB)
- Добавление задачи — НЕ работает (нет `onMutate`, задача теряется визуально)
- Toggle/Move/Delete задачи — НЕ работают (нет оптимистичных обновлений)
- Непосещённые разделы — пустые (данные загружаются только при навигации)
- Справочники (контексты, области, теги, проекты) — только чтение через кэш React Query

## Архитектурные решения

### Подход: Local-first

```
Dexie.js (IndexedDB) ← единственный источник истины
   ↕ useLiveQuery() — реактивные подписки в хуках
   ↕ SyncEngine — фоновая синхронизация с бекендом
Backend API — только приём/отдача данных, без состояния клиента
```

### Ключевые решения
- **Dexie.js** — локальная база данных на IndexedDB
- **Полностью убрать React Query** — все хуки подписываются на Dexie через `useLiveQuery`
- **LWW на уровне записи** — разрешение конфликтов по `updated_at` (последняя запись побеждает)
- **Клиентский UUID** — бекенд принимает `id` от клиента при создании сущностей
- **4 старые IDB базы → 1 Dexie БД** — консолидация всех хранилищ
- **Итеративная реализация** — сначала инфраструктура, потом задачи, потом справочники

### Ограничения useLiveQuery (важно!)

`useLiveQuery` из `dexie-react-hooks` — **только реактивная подписка** на IDB-запрос. Отличия от React Query:

| Возможность | React Query | useLiveQuery | Решение |
|---|---|---|---|
| Retry при ошибке | ✅ | ❌ | Нативный try/catch в компоненте |
| staleTime / gcTime | ✅ | ❌ | Не нужно — данные локальные |
| Prefetch | ✅ | ❌ | Прямая запись в Dexie при initial sync |
| Devtools | ✅ | ❌ | Dexie DevTools addon (chrome extension) |
| Background refetch | ✅ | ❌ | SyncEngine + online event |
| Loading/error state | ✅ (isLoading, error) | ✅ (возвращает undefined при загрузке) | Проверка на `undefined` |
| Suspense | ✅ | ❌ | Error boundary + loading component |

Мутации — через прямые вызовы `db.table.put()/delete()`, `useLiveQuery` автоматически обновит UI.

## LWW-стратегия разрешения конфликтов

### Принцип

При конфликте между локальным и серверным изменением одной записи:
1. Сравниваем `updatedAt` локальной и серверной версии
2. Если `server.updatedAt > local._lastSyncedAt` → берём серверную (она новее)
3. Если `local.updatedAt > server.updatedAt` → берём локальную (наша новее)
4. Если равны → берём серверную (приоритет сервера при равенстве)

### Когда возникают конфликты

Для **одноместного GTD-приложения** конфликты практически невозможны:
- Один пользователь = одно устройство обычно
- Даже при двух вкладках — Dexie работает в одном origin
- Риск появляется только при использовании с двух устройств одновременно

### Реализация

```typescript
// frontend/src/db/conflictResolution.ts

export async function mergeRecord<T extends { updatedAt: string; _lastSyncedAt: string | null }>(
  localRecord: T | undefined,
  serverRecord: T
): Promise<T> {
  if (!localRecord) return serverRecord

  // Локальная запись не была изменена — берём серверную
  if (localRecord._syncStatus === 'synced') return serverRecord

  // Локальная запись была изменена — сравниваем timestamps
  const localTime = new Date(localRecord.updatedAt).getTime()
  const serverTime = new Date(serverRecord.updatedAt).getTime()

  // Серверная новее — берём серверную
  if (serverTime > localTime) return serverRecord

  // Локальная новее или равна — оставляем локальную
  return localRecord
}
```

## SyncEngine — протокол

### Файл: `frontend/src/db/syncEngine.ts`

### Режимы синхронизации

| Режим | Когда | Что делает |
|-------|-------|------------|
| **Initial sync** | При логине | Pull всех данных с сервера → запись в Dexie |
| **Push** | При восстановлении сети | Отправка queued мутаций на сервер |
| **Pull** | По таймеру (каждые 5 мин) / online event | GET всех данных → merge с локальными |
| **SSE-driven** | При получении SSE-события | GET конкретной записи → merge |

### Initial sync (полный pull)

```
POST /api/auth/login → успешный логин

1. GET /api/tasks?limit=100&offset=0 → пока total > offset
2. GET /api/projects?limit=100&offset=0
3. GET /api/contexts?limit=100&offset=0
4. GET /api/areas?limit=100&offset=0
5. GET /api/tags?limit=100&offset=0
```

Пагинация: `limit=100`, циклический запрос пока `total > offset + limit`.

Для типичного GTD-приложения (< 1000 задач) — 1-10 запросов, < 2 секунды.

### Push (отправка мутаций)

```
1. SELECT * FROM mutations ORDER BY timestamp ASC
2. Для каждой мутации:
   a. POST/PUT/PATCH/DELETE на соответствующий эндпоинт
   b. При успехе → удалить из mutations, обновить _syncStatus='synced'
   c. При 409/422 → логировать, пропустить, increment retryCount
   d. При 401 → попытаться refresh token, повторить
   e. При network error → остановить push, ждать online
3. После всех мутаций → pull (обновить данные с сервера)
```

### Mapping: экшены → API-эндпоинты

| action | entityType | API вызов | Body |
|--------|-----------|-----------|------|
| create | task | POST /api/tasks | TaskCreate (с client-generated id) |
| update | task | PUT /api/tasks/{id} | TaskUpdate |
| toggle | task | PATCH /api/tasks/{id}/toggle | — |
| move | task | PATCH /api/tasks/{id}/move | { gtd_status } |
| delete | task | DELETE /api/tasks/{id} | — |
| create | project | POST /api/projects | ProjectCreate |
| update | project | PUT /api/projects/{id} | ProjectUpdate |
| delete | project | DELETE /api/projects/{id} | — |
| create | area | POST /api/areas | AreaCreate |
| update | area | PUT /api/areas/{id} | AreaUpdate |
| delete | area | DELETE /api/areas/{id} | — |
| create | context | POST /api/contexts | ContextCreate |
| update | context | PUT /api/contexts/{id} | ContextUpdate |
| delete | context | DELETE /api/contexts/{id} | — |
| create | tag | POST /api/tags | TagCreate |
| update | tag | PUT /api/tags/{id} | TagUpdate |
| delete | tag | DELETE /api/tags/{id} | — |

### Error handling

| Ошибка | Действие |
|--------|----------|
| 401 Unauthorized | Refresh token → повторить. Если не помогло → logout |
| 404 Not Found (при update/delete) | Запись удалена на сервере → удалить локально, удалить мутацию |
| 409 Conflict | Зависит от бекенда (не ожидается). Взять серверную версию |
| 422 Validation Error | Логировать, пропустить мутацию, increment retryCount. Не повторять автоматически |
| 500 Server Error | Повторить до 3 раз с экспоненциальной задержкой. Если не помогло — остановить push |
| Network Error | Остановить push, ждать online event |

## Стратегия ID (офлайн-создание)

### Проблема
Текущий бекенд генерирует ID сервером (`uuid.uuid4()`). При создании офлайн нужен ID до синхронизации.

### Решение
**Добавить на бекенде поддержку клиентского ID** — минимальное изменение:

```python
# backend/app/schemas/task.py
class TaskCreate(BaseModel):
    id: str | None = None          # ← добавить это поле
    title: str
    ...

# backend/app/services/task.py
async def create_task(self, ...):
    task = Task(
        id=data.id or str(uuid.uuid4()),  # ← использовать клиентский ID если есть
        ...
    )
```

Аналогично для Project, Area, Context, Tag.

На клиенте:
```typescript
import { v4 as uuidv4 } from 'uuid'  // или crypto.randomUUID()

const newTask = {
  id: crypto.randomUUID(),  // Генерируем на клиенте
  _syncStatus: 'local',
  ...
}
await db.tasks.put(newTask)
await db.mutations.add({ action: 'create', entityType: 'task', entityId: newTask.id, ... })
```

## Обработка спецсценариев

| Сценарий | Обработка |
|----------|-----------|
| **Создал офлайн → удалил до sync** | Проверка: если мутация `create` для entityId, а следом `delete` — удалить обе мутации, не отправлять на сервер |
| **Две вкладки одновременно** | Dexie автоматически обрабатывает `versionchange` события. `useLiveQuery` обновит данные при изменении из другой вкладки |
| **IndexedDB quota exceeded** | Ловить `QuotaExceededError`, показывать toast, прекращать sync до освобождения |
| **Logout → данные чужого пользователя** | `db.tasks.where('userId').equals(oldUserId).delete()` + очистить все таблицы при logout |
| **Бекенд изменил схему (миграция)** | Dexie version upgrade → при изменении версии — re-sync. Проверять версию API при initial sync |
| **Private browsing** | IndexedDB удаляется при закрытии. Acceptable — документировать. Приложение всё равно требует логин |
| **Rapid-мутации одной задачи** | При push — мержить мутации: если несколько update для одного entityId → отправить только последний |
| **Partial failure (5/10 мутаций)** | Обрабатывать по одной, не откатывать успешные. Остановить на первой ошибке 5xx |

## Миграция данных (старые IDB → Dexie)

```typescript
// frontend/src/db/migration.ts

export async function migrateOldData(): Promise<void> {
  const oldDatabases = [
    'todowka-query-cache',
    'todowka-cache',
    'todowka-local-changes',
    'todowka-offline',
  ]

  // Старые данные НЕ мигрируем — они потеряют актуальность при первом sync.
  // При логине initialSync загрузит свежие данные с сервера.
  // Старые базы просто удаляем.

  for (const name of oldDatabases) {
    await new Promise<void>((resolve) => {
      const req = indexedDB.deleteDatabase(name)
      req.onsuccess = () => resolve()
      req.onerror = () => resolve() // ignore errors
      req.onblocked = () => resolve() // ignore blocked
    })
  }
}
```
