# План v3: Local-first офлайн-режим на Dexie.js

**Дата:** 2026-04-18
**Версия:** 3.0 (исправленная по результатам двух критик)
**Статус:** Готово к реализации
**Выбранный вариант:** Вариант 3 — Глубокий рефакторинг (local-first на IndexedDB)

## Изменения относительно v2

| # | Проблема v2 | Решение v3 |
|---|-------------|------------|
| 1 | NotificationProvider + Trash используют useQueryClient напрямую | Добавлены шаги адаптации в итерации 1b и 2 |
| 2 | _syncStatus='deleted' конфликтует с физическим удалением | **Soft-delete**: задачи с `_syncStatus='deleted'` не удаляются физически, все запросы фильтруют через хелпер `activeRecords()` |
| 3 | RQ удаляется в 1b, но справочники ещё используют RQ | **RQ удаляется в итерации 4** — после того как все хуки переписаны |
| 4 | subtasksCount/subtasksCompleted — статичные числа | **Вычисляются динамически** через Dexie-запросы в useLiveQuery |
| 5 | tagIds → Tag[] маппинг для UI не описан | Добавлен mappers.ts с join из db.tags |
| 6 | Pull после прерванного push затирает pending мутации | Pull не мержит записи с pending mutations |
| 7 | Периодический pull — не описан orchestrator | Добавлен `SyncProvider` компонент |
| 8 | httpClient — неясно что остаётся | Уточнён список: refresh token, ApiError, 401 handling |
| 9 | Loading state из useLiveQuery | Добавлен хелпер `useDexieQuery` с isLoading |
| 10 | Существующие тесты мокают RQ | Добавлен шаг обновления тестов |
| 11 | crypto.randomUUID() требует HTTPS | Используется `uuid` npm-пакет |
| 12 | Subtasks — не описан Dexie-запрос | Уточнён запрос через parentTaskId |

## Постановка проблемы

Приложение должно полноценно работать при отсутствии подключения к бекенду. Текущее состояние:
- Редактирование задачи (PUT) — работает офлайн
- Добавление/Toggle/Move/Delete задачи — НЕ работают офлайн
- Непосещённые разделы — пустые
- Справочники — только чтение через кэш React Query

## Архитектурные решения

### Подход: Local-first

```
Dexie.js (IndexedDB) ← единственный источник истины
   ↕ useLiveQuery() — реактивные подписки в хуках
   ↕ SyncEngine — фоновая синхронизация с бекендом
Backend API — только приём/отдача данных
```

### Ключевые решения
- **Dexie.js** — локальная база данных на IndexedDB
- **React Query удаляется в итерации 4** — до этого RQ и Dexie работают параллельно; хуки переписываются по одному
- **LWW на уровне записи** — по `updated_at`
- **Клиентский UUID** через npm-пакет `uuid` — бекенд принимает `id` от клиента
- **Soft-delete** — `_syncStatus='deleted'`, фильтрация через хелпер
- **Динамические вычисления** — subtasksCount, GtdCounts вычисляются из Dexie, не хранятся

```typescript
// frontend/src/db/database.ts (продолжение)

export function activeTasks(userId: string) {
  return db.tasks
    .where('[userId+gtdStatus]')
    .between(
      [userId, Dexie.minKey],
      [userId, Dexie.maxKey]
    )
    .filter(t => t._syncStatus !== 'deleted')
}

export function activeTasksByStatus(userId: string, status: string) {
  return db.tasks
    .where('[userId+gtdStatus]')
    .equals([userId, status])
    .filter(t => t._syncStatus !== 'deleted')
}

export function activeTasksByProject(userId: string, projectId: string) {
  return db.tasks
    .where('[userId+projectId]')
    .equals([userId, projectId])
    .filter(t => t._syncStatus !== 'deleted')
}

export function activeTable<T extends { _syncStatus: SyncStatus }>(
  table: Table<T>,
  userId: string
) {
  return table
    .where('userId')
    .equals(userId)
    .filter(r => r._syncStatus !== 'deleted')
}
```

## Mappers: DbTask ↔ UI Task

### Файл: `frontend/src/db/mappers.ts`

```typescript
import { db, type DbTask } from './database'
import type { Tag } from '../hooks/useTags'

// UI-тип задачи (то что видят компоненты)
// Совпадает с текущим интерфейсом Task для совместимости
export interface UiTask {
  id: string
  title: string
  description: string | null
  completed: boolean
  gtd_status: string
  context_id: string | null
  area_id: string | null
  project_id: string | null
  project: { id: string; name: string; color: string | null; is_active: boolean } | null
  context: { id: string; name: string; color: string | null; icon: string | null } | null
  parent_task_id: string | null
  position: number
  due_date: string | null
  notes: string | null
  recurrence_type: string | null
  recurrence_config: unknown
  recurrence_end_date: string | null
  reminder_time: string | null
  reminder_offsets: number[] | null
  reminder_fired: boolean
  is_recurring: boolean
  tags: Tag[]
  subtasks_count: number
  subtasks_completed: number
  user_id: string
  created_at: string
  updated_at: string
}

// DbTask → UiTask (для UI)
// Выполняет join: tagIds → tags[], вычисляет subtasks_count
export async function dbTaskToUi(task: DbTask): Promise<UiTask> {
  // Join tagIds → Tag[]
  let tags: Tag[] = []
  if (task.tagIds.length > 0) {
    const tagRecords = await db.tags.where('id').anyOf(task.tagIds).toArray()
    tags = tagRecords
      .filter(t => t._syncStatus !== 'deleted')
      .map(t => ({ id: t.id, name: t.name, color: t.color }))
  }

  // Вычислить subtasks_count динамически
  const subtasks = await db.tasks
    .where('parentTaskId')
    .equals(task.id)
    .filter(t => t._syncStatus !== 'deleted')
    .toArray()
  const subtasksCount = subtasks.length
  const subtasksCompleted = subtasks.filter(s => s.isCompleted).length

  // Join project
  let project = null
  if (task.projectId) {
    const p = await db.projects.get(task.projectId)
    if (p && p._syncStatus !== 'deleted') {
      project = { id: p.id, name: p.name, color: p.color, is_active: p.isActive }
    }
  }

  // Join context
  let context = null
  if (task.contextId) {
    const c = await db.contexts.get(task.contextId)
    if (c && c._syncStatus !== 'deleted') {
      context = { id: c.id, name: c.name, color: c.color, icon: c.icon }
    }
  }

  return {
    id: task.id,
    title: task.title,
    description: task.description,
    completed: task.isCompleted,
    gtd_status: task.gtdStatus,
    context_id: task.contextId,
    area_id: task.areaId,
    project_id: task.projectId,
    project,
    context,
    parent_task_id: task.parentTaskId,
    position: task.position,
    due_date: task.dueDate,
    notes: task.notes,
    recurrence_type: task.recurrenceType,
    recurrence_config: task.recurrenceConfig ? JSON.parse(task.recurrenceConfig) : null,
    recurrence_end_date: task.recurrenceEndDate,
    reminder_time: task.reminderTime,
    reminder_offsets: task.reminderOffsets ? JSON.parse(task.reminderOffsets) : null,
    reminder_fired: task.reminderFired,
    is_recurring: task.isRecurring,
    tags,
    subtasks_count: subtasksCount,
    subtasks_completed: subtasksCompleted,
    user_id: task.userId,
    created_at: task.createdAt,
    updated_at: task.updatedAt,
  }
}

// API-ответ → DbTask (для записи в Dexie при pull)
export function apiTaskToDb(
  apiTask: Record<string, unknown>,
  userId: string
): DbTask {
  const tagIds = (apiTask.tags as { id: string }[] ?? []).map(t => t.id)
  return {
    id: apiTask.id as string,
    userId,
    title: apiTask.title as string,
    description: (apiTask.description as string | null) ?? null,
    isCompleted: (apiTask.is_completed as boolean) ?? false,
    completedAt: (apiTask.completed_at as string | null) ?? null,
    gtdStatus: (apiTask.gtd_status as string) ?? 'inbox',
    contextId: (apiTask.context_id as string | null) ?? null,
    areaId: (apiTask.area_id as string | null) ?? null,
    projectId: (apiTask.project_id as string | null) ?? null,
    parentTaskId: (apiTask.parent_task_id as string | null) ?? null,
    position: (apiTask.position as number) ?? 0,
    dueDate: (apiTask.due_date as string | null) ?? null,
    notes: (apiTask.notes as string | null) ?? null,
    recurrenceType: (apiTask.recurrence_type as string | null) ?? null,
    recurrenceConfig: apiTask.recurrence_config ? JSON.stringify(apiTask.recurrence_config) : null,
    recurrenceEndDate: (apiTask.recurrence_end_date as string | null) ?? null,
    reminderTime: (apiTask.reminder_time as string | null) ?? null,
    reminderOffsets: apiTask.reminder_offsets ? JSON.stringify(apiTask.reminder_offsets) : null,
    reminderFired: (apiTask.reminder_fired as boolean) ?? false,
    isRecurring: (apiTask.is_recurring as boolean) ?? false,
    tagIds,
    trashedAt: null,
    createdAt: (apiTask.created_at as string) ?? new Date().toISOString(),
```

## Хелпер для useLiveQuery с loading state

### Файл: `frontend/src/db/hooks.ts`

```typescript
import { useLiveQuery } from 'dexie-react-hooks'
import { useState, useEffect } from 'react'

// useLiveQuery возвращает undefined при загрузке.
// Этот хелпер даёт привычный isLoading.
export function useDexieQuery<T>(querier: () => T | Promise<T>, deps?: unknown[]) {
  const data = useLiveQuery(querier, deps)
  const [isLoading, setIsLoading] = useState(true)

  useEffect(() => {
    if (data !== undefined) setIsLoading(false)
  }, [data])

  return { data: data as T | undefined, isLoading }
}

// Хелпер для online/offline статуса
export function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(navigator.onLine)

  useEffect(() => {
    const on = () => setIsOnline(true)
    const off = () => setIsOnline(false)
    window.addEventListener('online', on)
    window.addEventListener('offline', off)
    return () => {
      window.removeEventListener('online', on)
      window.removeEventListener('offline', off)
    }
  }, [])

  return isOnline
}
```

## LWW-стратегия разрешения конфликтов

### Принцип

При конфликте между локальным и серверным изменением одной записи:
1. Если локальная `_syncStatus === 'synced'` → берём серверную
2. Если локальная `_syncStatus !== 'synced'` → сравниваем `updatedAt`:
   - Серверная новее → берём серверную
   - Локальная новее или равна → оставляем локальную
3. **Pull не мержит записи с pending mutations** — проверяем `db.mutations.where('[entityType+entityId]')`

### Реализация

```typescript
// frontend/src/db/conflictResolution.ts

import type { SyncStatus } from './database'

interface Mergeable {
  updatedAt: string
  _syncStatus: SyncStatus
  _lastSyncedAt: string | null
}

export async function shouldSkipMerge(
  entityType: string,
  entityId: string
): Promise<boolean> {
  const pending = await db.mutations
    .where('[entityType+entityId]')
    .equals([entityType, entityId])
    .count()
  return pending > 0
}

export function mergeRecord<T extends Mergeable>(
  localRecord: T | undefined,
  serverRecord: T
): T {
  if (!localRecord) return serverRecord
  if (localRecord._syncStatus === 'synced') return serverRecord

  const localTime = new Date(localRecord.updatedAt).getTime()
  const serverTime = new Date(serverRecord.updatedAt).getTime()

  if (serverTime > localTime) return serverRecord
  return localRecord
}
```

## SyncEngine — протокол

### Режимы синхронизации

| Режим | Когда | Что делает |
|-------|-------|------------|
| **Initial sync** | При логине | Pull всех данных → Dexie |
| **Push** | Online event + после initial sync | Отправка queued мутаций |
| **Pull** | Каждые 15 мин / после push | GET всех данных → merge |
| **SSE-driven** | SSE-событие | GET конкретной записи → merge |

### SyncProvider — оркестратор

Компонент-обёртка, запускается в `AppLayout` (только для authenticated пользователей):

```typescript
// frontend/src/components/SyncProvider.tsx

export function SyncProvider({ children }: { children: React.ReactNode }) {
  // 1. При mount → запустить SyncEngine (periodic pull + push on online)
  // 2. При unmount → остановить таймеры
  // 3. Предоставляет sync status через context:
  //    - isSyncing, pendingCount, lastSyncAt, isOnline
  return <SyncContext.Provider value={...}>{children}</SyncContext.Provider>
}
```

### Initial sync

```
POST /api/auth/login → успешный логин

1. GET /api/tasks?limit=100&offset=0 → цикл пока total > offset
2. GET /api/projects?limit=100&offset=0
3. GET /api/contexts?limit=100&offset=0
4. GET /api/areas?limit=100&offset=0
5. GET /api/tags?limit=100&offset=0

Каждую запись → mergeRecord → db.table.put()
Skip если shouldSkipMerge (pending mutations)
```

### Push

```
1. SELECT * FROM mutations ORDER BY timestamp ASC
2. Оптимизация: мержить мутации одного entityId
   (несколько update → отправить последний; create+delete → удалить обе)
3. Для каждой мутации:
   a. POST/PUT/PATCH/DELETE
   b. Успех → удалить из mutations, обновить _syncStatus='synced'
   c. 401 → refresh token, повторить
   d. 404 → удалить локально + удалить мутацию
   e. 422 → логировать, пропустить
   f. 500 → retry до 3 раз, затем остановить
   g. Network error → остановить push
4. После push → pull
```

### Error handling

| Ошибка | Действие |
|--------|----------|
| 401 | Refresh token → повторить. Не помогло → logout |
| 404 | Запись удалена на сервере → удалить локально + мутацию |
| 409 | Взять серверную версию |
| 422 | Логировать, пропустить, не повторять |
| 500 | Retry ×3 с backoff, затем остановить |
| Network Error | Остановить push, ждать online |
| Partial JSON | TypeError → retry через 30с |

## Обработка спецсценариев

| Сценарий | Обработка |
|----------|-----------|
| **Создал офлайн → удалил до sync** | При push: найти create + delete для одного entityId → удалить обе мутации, удалить запись из Dexie |
| **Две вкладки** | Dexie `versionchange` + `useLiveQuery` — автоматическая синхронизация |
| **IndexedDB quota exceeded** | Ловить `QuotaExceededError`, toast, прекратить sync |
| **Logout** | Очистить все таблицы Dexie по userId: `db.tasks.where('userId').equals(uid).delete()` + projects, areas, contexts, tags, mutations, syncMeta |
| **Бекенд-миграция** | Dexie version upgrade → re-sync |
| **Private browsing** | Acceptable — документировать |
| **Rapid-мутации** | Мержить при push: несколько update → последний |
| **Partial failure** | Обрабатывать по одной, не откатывать успешные |
| **Toggle recurring офлайн** | Бекенд при toggle создаст следующую задачу. Локально пользователь увидит её после sync. Acceptable |
| **Initial sync при плохом соединении** | Retry с экспоненциальной задержкой. Показать loading screen до завершения |

## httpClient — что остаётся после упрощения

**Остаётся:**
- JWT Auth (access token из authStore)
- 401 → refresh token → retry
- ApiError класс
- Все HTTP-методы (get/post/put/patch/delete)
- Debug логирование

**Убирается:**
- Offline queue (`queueMutationFn`) → заменяет mutations table
- GET cache (`getCache/setCache`) → заменяет Dexie
- Online/offline toasts → заменяет SyncProvider + SyncStatus
- `OfflineQueueError` класс → не нужен
- `BACKEND_RECOVERED` handler → SyncProvider
- `hasShownOfflineToast` / `hasShownQueueToast` flags → SyncProvider

## Миграция данных

```typescript
// frontend/src/db/migration.ts
export async function migrateOldData(): Promise<void> {
  const oldDatabases = [
    'todowka-query-cache',
    'todowka-cache',
    'todowka-local-changes',
    'todowka-offline',
  ]
  for (const name of oldDatabases) {
    await new Promise<void>((resolve) => {
      const req = indexedDB.deleteDatabase(name)
      req.onsuccess = () => resolve()
      req.onerror = () => resolve()
      req.onblocked = () => resolve()
    })
  }
}
```

Старые данные НЕ мигрируются — при логине initialSync загрузит свежие данные с сервера.
