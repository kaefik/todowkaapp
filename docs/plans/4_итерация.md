## Фича 1. Weekly Review Wizard

### Концепция

Пошаговый wizard на `/review` для проведения еженедельного обзора по GTD-методологии. Три шага, каждый фокусируется на отдельном аспекте:

1. **Обработка Inbox** — просмотр всех задач во Inbox, быстрое распределение (действие? проект? когда-нибудь? мусор?)
2. **Проверка проектов** — список активных проектов, индикатор наличия next action для каждого
3. **Ревизия Someday/Maybe** — просмотр задач в Someday, возможность продвинуть или архивировать

### Backend

#### 1.1. Новый API endpoint: `GET /api/review/status`

Возвращает агрегированные данные для weekly review:

```python
# backend/app/api/review.py

@router.get("/status")
async def get_review_status(current_user, db):
    return {
        "inbox_count": ...,          # кол-во задач в inbox
        "inbox_tasks": [...],         # задачи inbox
        "active_projects": [...],     # проекты с has_next_action флагом
        "someday_tasks": [...],       # задачи Someday/Maybe
        "last_review_date": ...,      # дата последнего обзора (из preferences)
        "review_count": ...,          # сколько обзоров проведено
    }
```

#### 1.2. Новый API endpoint: `POST /api/review/complete`

Фиксирует завершение weekly review:

- Сохраняет дату последнего обзора в пользовательских настройках
- Увеличивает счётчик проведённых обзоров
- Возвращает подтверждение

#### 1.3. Модель: расширение User

Добавить поля в модель User:
- `last_review_at: datetime | None` — дата последнего weekly review
- `review_count: int = 0` — общее количество проведённых обзоров

#### 1.4. Service: `ReviewService`

Инкапсулирует логику агрегации данных для review:
- `get_review_status(user_id, db)` — собирает данные из Task, Project
- `complete_review(user_id, db)` — обновляет last_review_at, review_count

### Frontend

#### 1.5. Маршрут `/review`

Новый маршрут в `router.tsx`:
- Защищённый (требует авторизацию)
- Полноэкранный wizard layout (как OnboardingWizard)
- Sidebar скрыт, показывается progress bar

#### 1.6. Компоненты wizard

```
src/
├── routes/
│   └── Review.tsx              # Страница-контейнер, state machine шагов
├── components/
│   └── review/
│       ├── ReviewWizard.tsx     # Wrapper с progress bar, навигацией шагов
│       ├── ReviewInbox.tsx      # Шаг 1: обработка Inbox
│       ├── ReviewProjects.tsx   # Шаг 2: проверка проектов
│       └── ReviewSomeday.tsx    # Шаг 3: ревизия Someday/Maybe
```

#### 1.7. ReviewInbox (Шаг 1)

- Список всех задач в Inbox с быстрыми действиями:
  - «Сделать» → перемещает в Active (next action)
  - «Проект» → создаёт/привязывает к проекту
  - «Когда-нибудь» → перемещает в Someday
  - «Удалить» → в корзину
- Счётчик обработанных / оставшихся
- Индикатор «Inbox до нуля» когда все обработаны

#### 1.8. ReviewProjects (Шаг 2)

- Список активных проектов
- Для каждого проекта: название, кол-во задач, есть ли хотя бы одна с `gtd_status='active'`
- Красный индикатор если у проекта нет next action
- Быстрое действие: «Добавить next action» → mini-form создания задачи в проект

#### 1.9. ReviewSomeday (Шаг 3)

- Список задач в Someday/Maybe
- Для каждой: быстрые действия
  - «Активировать» → перемещает в Active
  - «В проект» → перемещает в проект
  - «Оставить» → без изменений
  - «В корзину» → trashed

#### 1.10. Завершение review

- Экран «Обзор завершён» с краткой статистикой:
  - Обработано Inbox задач
  - Проектов без next action
  - Someday задач активировано
- Кнопка «На главную» → редирект на `/`
- Сохранение через `POST /api/review/complete`

#### 1.11. Напоминание о weekly review

- На главной странице (или в sidebar) — индикатор «Время для еженедельного обзора»
- Показывается если `last_review_at` был более 7 дней назад
- Ссылка на `/review`

## Фича 2. Управление сессиями

### Концепция

Пользователь видит все активные сессии (устройства, даты входа), может отозвать отдельную сессию или выйти со всех устройств. Текущая сессия помечается визуально.

### Backend

#### 2.1. Новая модель: `Session`

```python
# backend/app/models/session.py

class Session(Base):
    __tablename__ = "sessions"

    id: Mapped[str]          # UUID, PK
    user_id: Mapped[int]     # FK -> users.id
    refresh_token_jti: Mapped[str]   # JTI refresh-токена
    user_agent: Mapped[str | None]   # браузер/устройство
    ip_address: Mapped[str | None]   # IP входа
    created_at: Mapped[datetime]     # дата создания сессии
    last_activity: Mapped[datetime]  # последнее использование
    is_current: не хранится, вычисляется на лету
```

Индексы:
- `ix_sessions_user_id` — (user_id)
- `ix_sessions_refresh_token_jti` — (refresh_token_jti, unique)

#### 2.2. API endpoints

```python
# backend/app/api/sessions.py

GET    /api/sessions           # список активных сессий
DELETE /api/sessions/{id}      # отзыв конкретной сессии
DELETE /api/sessions           # «выйти со всех устройств» (кроме текущей)
```

#### 2.3. Интеграция в auth flow

- **Login**: создавать запись в `sessions` таблице
- **Refresh**: обновлять `last_activity` в записи сессии
- **Logout**: удалять запись сессии + отзывать refresh token
- **Смена пароля**: удалять ВСЕ записи сессий пользователя кроме текущей (уже частично реализовано через `password_changed_at`)

#### 2.4. Парсинг User-Agent

Использовать библиотеку `user-agents` (или простой парсинг) для извлечения:
- Браузер + версия
- ОС + версия
- Устройство (desktop/mobile/tablet)

Хранить в `user_agent` как JSON или отдельные поля.

#### 2.5. Service: `SessionService`

```python
class SessionService:
    async def create_session(user_id, refresh_jti, user_agent, ip) -> Session
    async def get_user_sessions(user_id, current_jti) -> list[SessionResponse]
    async def revoke_session(session_id, user_id) -> None
    async def revoke_all_sessions(user_id, current_jti) -> None
    async def update_activity(refresh_jti) -> None
    async def cleanup_expired() -> None  # фоновая очистка
```

### Frontend

#### 2.6. Страница `/settings/sessions`

Или секция внутри `/settings`:
- Список активных сессий в виде карточек
- Текущая сессия помечена бейджем «Текущая»
- Каждая карточка: иконка устройства, браузер, ОС, IP, дата входа, дата последней активности
- Кнопка «Отозвать» на каждой карточке (кроме текущей)
- Кнопка «Выйти со всех устройств» внизу (с confirm dialog)

#### 2.7. Компоненты

```
src/components/
└── sessions/
    ├── SessionList.tsx       # Список сессий
    ├── SessionCard.tsx       # Карточка одной сессии
    └── RevokeAllButton.tsx   # Кнопка «выйти со всех»
```

#### 2.8. Определение текущей сессии

- При логине backend возвращает `session_id` в response (в httpOnly cookie или отдельном поле)
- Frontend сохраняет `session_id` в memory (Zustand store)
- При отображении списка — сравниваем с текущим session_id

## Фича 3. Анимации (Polish)

### Концепция

Добавить плавные анимации переходов, mount/unmount эффекты для модалок и списков. Использовать **CSS-only подход** + React Transition Group (без Framer Motion — тяжелая зависимость для polish).

### Подход

**Не добавлять** тяжелые библиотеки анимаций. Вместо этого:
- CSS `@keyframes` для mount/unmount
- CSS `transition` для интерактивных элементов (уже частично используется)
- View Transitions API для page transitions (современные браузеры)
- `@media (prefers-reduced-motion: reduce)` для доступности

### Frontend

#### 3.1. Базовые CSS анимации

Добавить в `index.css` набор utility-анимаций:

```css
@keyframes fadeIn {
  from { opacity: 0; }
  to { opacity: 1; }
}

@keyframes slideUp {
  from { opacity: 0; transform: translateY(8px); }
  to { opacity: 1; transform: translateY(0); }
}

@keyframes slideDown {
  from { opacity: 0; transform: translateY(-8px); }
  to { opacity: 1; transform: translateY(0); }
}

@keyframes scaleIn {
  from { opacity: 0; transform: scale(0.95); }
  to { opacity: 1; transform: scale(1); }
}

.animate-fade-in { animation: fadeIn 200ms ease-out; }
.animate-slide-up { animation: slideUp 200ms ease-out; }
.animate-slide-down { animation: slideDown 200ms ease-out; }
.animate-scale-in { animation: scaleIn 150ms ease-out; }
```

#### 3.2. Reduced motion

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

#### 3.3. Page transitions через View Transitions API

Обёртка в router для плавных переходов между страницами:

```tsx
// Условное использование, fallback для неподдерживающих браузеров
if (document.startViewTransition) {
  document.startViewTransition(() => {
    // navigate
  });
} else {
  // navigate directly
}
```

#### 3.4. Modal animations

- TaskDetailModal, TaskEditModal, ConfirmDialog, DeleteAccountModal — добавить `animate-scale-in`
- Overlay: `animate-fade-in`
- Закрытие: CSS transition opacity → 0 + transform scale(0.95)

#### 3.5. List item animations

- Задачи в списках: `animate-slide-up` при появлении с `animation-delay` для stagger-эффекта
- Новая задача вверху списка: `animate-slide-down`

#### 3.6. Sidebar toggle

- Плавная анимация ширины sidebar через CSS `transition: width 200ms ease`
- Overlay fade in/out

## Фича 5. Rate Limiting на все API

### Концепция

Расширить rate limiting с login/register на все API-эндпоинты. Использовать тот же slowapi, который уже интегрирован.

### Backend

#### 5.1. Глобальный rate limiter

Добавить middleware или dependency-level rate limiting:

```python
# backend/app/main.py или dependencies.py

# Глобальный лимит: 60 запросов/минута на пользователя
limiter.limit("60/minute")(get_current_user)

# Публичные endpoints: 30 запросов/минута на IP
limiter.limit("30/minute")(public_endpoints)
```

#### 5.2. Категории лимитов

| Категория | Лимит | Эндпоинты |
|-----------|-------|-----------|
| Auth (строгий) | 3/мин | login, register |
| Write operations | 20/мин | POST/PUT/PATCH/DELETE всех ресурсов |
| Read operations | 60/мин | GET всех ресурсов |
| SSE connections | 5/мин | `/api/sse/*` |
| Export/Import | 5/мин | `/api/export-import/*` |

#### 5.3. Конфигурация через .env

```env
RATE_LIMIT_AUTH=3/minute
RATE_LIMIT_WRITE=20/minute
RATE_LIMIT_READ=60/minute
RATE_LIMIT_SSE=5/minute
RATE_LIMIT_EXPORT=5/minute
```

#### 5.4. Интеграция

- Создать `backend/app/rate_limit.py` с конфигурацией
- Применить через FastAPI dependencies на роутерах
- Логирование 429 ответов (уже есть кастомный handler)
- Фронтенд уже обрабатывает 429 (показывает локализованные сообщения)

## Фича 6. Security Audit

### Концепция

Провести структурированный аудит безопасности, исправить найденные проблемы, добавить превентивные меры.

### Этапы

#### 6.1. Код-ревью безопасности (manual)

Проверить:
- [ ] Все API endpoints используют авторизацию (нет «голых» роутов)
- [ ] SQL injection — все запросы через SQLAlchemy ORM (параметризованные)
- [ ] XSS — React по умолчанию экранирует, проверить `dangerouslySetInnerHTML`
- [ ] CSRF — cookie-only модель + SameSite=lax (реализовано в предыдущих итерациях)
- [ ] Path traversal — проверить работу с файлами если есть
- [ ] Информация в error responses — не утечка ли stack traces, путей, SQL

#### 6.2. Зависимости

- [ ] `pip audit` — проверить Python зависимости на известные CVE
- [ ] `npm audit` — проверить JS зависимости
- [ ] Обновить пакеты с критическими уязвимостями

#### 6.3. JWT Security

- [ ] Секретный ключ — проверить что не хардкод, берётся из env
- [ ] Алгоритм — убедиться что HS256 (не none)
- [ ] Токен expiry — access: 15 мин, refresh: 7 дней (проверить)
- [ ] Token rotation — при refresh старый refresh token отзывается
- [ ] Отозванные токены — проверить cleanup (не накапливаются ли в БД)

#### 6.4. Cookie Security

- [ ] httpOnly — оба токена (access + refresh)
- [ ] Secure flag — в production (не в dev)
- [ ] SameSite — lax (проверить)
- [ ] Path — access: `/`, refresh: `/api/auth` (проверить)

#### 6.5. Data validation

- [ ] Pydantic схемы — все input валидируются
- [ ] File upload — лимит размера если есть
- [ ] Pagination — лимит на page size (предотвратить DoS)

#### 6.6. Cleanup: RevokedToken таблица

- [ ] Добавить фоновую задачу для очистки истёкших revoked tokens
- [ ] Запускать раз в сутки через APScheduler (уже есть в lifespan)
- [ ] Удалять записи старше 7 дней

#### 6.7. Security Headers

Добавить middleware для security headers:

```python
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 0  # устаревший, но для совместимости
Referrer-Policy: strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'; ...
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

## Риски и mitigation

| Риск | Вероятность | Влияние | Mitigation |
|------|-------------|---------|------------|
| Weekly Review wizard слишком сложный | Средняя | Среднее | Начать с MVP: 3 простых шага без излишнего UI |
| Session tracking ломает текущий auth flow | Низкая | Высокое | Писать поверх существующего, не переписывать auth |
| View Transitions API не поддерживается | Высокая | Низкое | Graceful fallback — обычная навигация без анимации |
| Rate limiting слишком агрессивный | Средняя | Среднее | Настраиваемые лимиты через .env, начать с щедрых значений |
| Security headers ломают frontend | Низкая | Среднее | Тестировать CSP постепенно, начать с report-only |

## Критерии готовности

Каждая фича считается завершённой когда:

1. **SQL-индексы** — миграция применена, запросы к projects/areas/contexts используют индексы (EXPLAIN QUERY PLAN)
2. **Rate limiting** — все API покрыты лимитами, 429 корректно обрабатывается frontend
3. **Управление сессиями** — можно увидеть свои сессии, отозвать одну или все
4. **Weekly Review** — можно пройти все 3 шага wizard, данные сохраняются
5. **Анимации** — модалки плавно появляются, списки с stagger, reduced motion работает
6. **Security audit** — все чеклисты пройдены, нет критических уязвимостей
7. **docs/features.md** обновлён
8. **Все линтеры и тесты проходят** (`ruff check`, `npm run lint`, `npx tsc --noEmit`)
