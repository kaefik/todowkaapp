# Анализ фич Итерации 2 — GTD-методология

---

## Обзор

Документ содержит всесторонний анализ всех функций, запланированных во Итерации 2 для внедрения GTD-методологии в Todowka.

---

## Фичи (от самых простых к самым сложным)

### 1. 🔴 Контексты (Contexts)

**Суть:** Классификация задач по месту/обстоятельствам выполнения (Дом, Работа, Звонки, Компьютер).

**Сложность:** ⭐⭐ (2/5)

**Техническая реализация:**
- БД: новая таблица `contexts` с полями (id, user_id, name, color, icon)
- API: 4 CRUD endpoints
- Frontend: UI управления + dropdown в форме задачи

**ЗА:**
- ✓ Ключевой элемент GTD — помогает выполнять задачи в правильном контексте
- ✓ Простая реализация, низкий риск багов
- ✓ Быстрая отдача ценности (value)
- ✓ Хорошая визуализация (цвета + иконки)
- ✓ Минимальная нагрузка на UI

**ПРОТИВ:**
- ✗ Может быть избыточным для простых пользователей
- ✗ Требует дисциплины от пользователя для заполнения
- ✗ Добавляет ещё один уровень сложности в UI

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — базовый элемент GTD, прост в реализации, высокая ценность

---

### 2. 🔴 Области (Areas)

**Суть:** Высокоуровневые категории жизненных сфер (Здоровье, Финансы, Карьера, Семья).

**Сложность:** ⭐⭐ (2/5)

**Техническая реализация:**
- БД: новая таблица `areas`
- API: 4 CRUD endpoints
- Frontend: UI управления + dropdown

**ЗА:**
- ✓ Помогает держать фокус на важном
- ✓ Связывает задачи и проекты в осмысленные группы
- ✓ Полезно для периодического обзора (Weekly Review)
- ✓ Очень простая реализация

**ПРОТИВ:**
- ✗ Меньшая видимость в UI (менее очевидная польза)
- ✗ Может дублировать проекты или теги
- ✗ Не всем пользователям нужен такой уровень абстракции

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — важен для GTD, прост, дополняет проекты

---

### 3. 🟡 Теги (Tags)

**Суть:** Гибкая many-to-many классификация задач (срочно, важно, идея, делегировать).

**Сложность:** ⭐⭐⭐ (3/5)

**Техническая реализация:**
- БД: таблицы `tags` + `task_tags` (M:N)
- API: 6 endpoints (включая привязку/отвязку)
- Frontend: multi-select компонент, chips в списке

**ЗА:**
- ✓ Максимальная гибкость классификации
- ✓ Можно комбинировать с другими фильтрами
- ✓ Полезно для быстрого поиска
- ✓ Стандартная UX-паттерна

**ПРОТИВ:**
- ✗ More-to-many сложнее чем простые FK
- ✗ Риск хаоса в тегах (дублирование, неструктурированность)
- ✗ Требует особого UI для multi-select
- ✗ Возможны N+1 запросы без оптимизации

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — стандартная фича, нужна для гибкости

---

### 4. 🟡 Inbox — быстрый захват задач

**Суть:** Быстрое создание задач без подробностей, с последующей обработкой.

**Сложность:** ⭐⭐⭐ (3/5)

**Техническая реализация:**
- BE: default `gtd_status=inbox`, endpoint `/clarify`
- FE: Quick Capture Bar, inline-форма уточнения
- Клавиатурный шорткат

**ЗА:**
- ✓ Фундамент GTD — "capture everything"
- ✓ Психологически облегчает добавление задач
- ✓ Шорткат повышает продуктивность
- ✓ Reduces friction

**ПРОТИВ:**
- ✗ Может создать плохую привычку (задачи без контекста)
- ✗ Требует дисциплины — регулярно обрабатывать Inbox
- ✗ Дополнительные UI-компоненты (Quick Capture Bar)

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — фундамент GTD, важен для UX

---

### 5. 🟡 GTD-статусы и Workflow

**Суть:** Канбан-подобный статусный поток задач (Inbox → Next Action → Waiting → Someday → Completed).

**Сложность:** ⭐⭐⭐ (3/5)

**Техническая реализация:**
- БД: поле `gtd_status` в tasks
- API: endpoints `/move`, `/reorder`
- FE: 5 отдельных страниц + навигация

**ЗА:**
- ✓ Ядро GTD-методологии
- ✓ Интуитивный визуальный поток
- ✓ Помогает приоритизировать
- ✓ Стандартная паттерна для todo-приложений

**ПРОТИВ:**
- ✗ Может быть избыточным для простых пользователей
- ✗ Требует обучения (какой статус когда использовать)
- ✗ Усложняет навигацию (5 страниц вместо 1)
- ✗ Риск забытых задач в Inbox

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — это суть GTD-приложения

---

### 6. 🟡 Подзадачи (Subtasks)

**Суть:** Иерархические задачи — разбивка больших задач на мелкие шаги.

**Сложность:** ⭐⭐⭐ (3/5)

**Техническая реализация:**
- БД: `parent_task_id` FK → self-reference
- API: endpoints для subtasks
- FE: nested UI, индикатор прогресса (3/5)

**ЗА:**
- ✓ Очень полезно для крупных задач
- ✓ Визуализация прогресса мотивирует
- ✓ Стандартная фича для todo-приложений
- ✓ Улучшает UX для сложных задач

**ПРОТИВ:**
- ✗ Self-reference сложнее обычных FK
- ✗ Глубокая вложенность — проблемы с UI
- ✗ Вопрос: что делать с родительской задачей при завершении подзадач?
- ✗ Может усложнить фильтрацию

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — ожидаемая фича, полезная

---

### 7. 🟠 Проекты (Projects)

**Суть:** Группировка задач по проектам с прогресс-баром и привязкой к областям.

**Сложность:** ⭐⭐⭐⭐ (4/5)

**Техническая реализация:**
- БД: таблица `projects` + FK от tasks + FK от areas
- API: 5 endpoints + подсчёт прогресса
- FE: страница проектов, детализация проекта, прогресс-бар

**ЗА:**
- ✓ Ключевой элемент GTD — "projects are planned outcomes"
- ✓ Связывает задачи в осмысленные группы
- ✓ Прогресс-бар мотивирует
- ✓ Связь с областями даёт целостную картину

**ПРОТИВ:**
- ✗ Сложнее чем простые теги/контексты
- ✗ Требует расчёта прогресса (запросы к БД)
- ✗ Дополнительные страницы и UI
- ✗ Может дублировать области
- ✗ Вопрос: удалять ли задачи при удалении проекта?

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — важный элемент GTD

---

### 8. 🟠 Фильтрация и поиск

**Суть:** Комбинированные фильтры + полнотекстовый поиск по задачам.

**Сложность:** ⭐⭐⭐⭐ (4/5)

**Техническая реализация:**
- BE: `TaskFilterService`, динамические query
- 10+ query params в `/api/tasks`
- FE: панель фильтров, поиск с debounce, сортировка
- URL state management

**ЗА:**
- ✓ Критично для большого количества задач
- ✓ Повышает продуктивность (быстро находить нужное)
- ✓ Комбинирование фильтров — мощная фича
- ✓ Shareable links через URL

**ПРОТИВ:**
- ✗ Сложная логика построения query
- ✗ Риск багов при пересечении фильтров
- ✗ Перегрузка UI (много dropdown'ов)
- ✗ Поиск может быть медленным без FTS5
- ✗ Дебоунс — дополнительная сложность

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — базовая потребность для usability

---

### 9. 🟠 Обновление формы задачи

**Суть:** Расширение формы всеми новыми полями (GTD-статус, контекст, проект, область, дедлайн, теги, заметки).

**Сложность:** ⭐⭐⭐⭐ (4/5)

**Техническая реализация:**
- FE: 8+ новых полей в форме
- React Hook Form + Zod обновление
- Много dropdown'ов с загрузкой данных из API
- Progressive disclosure (раскрытие по необходимости)

**ЗА:**
- ✓ Один интерфейс для всех настроек задачи
- ✓ Централизованная логика

**ПРОТИВ:**
- ✗ Форма становится огромной
- ✗ UX перегрузка — слишком много полей
- ✗ Сложность валидации
- ✗ Нужно загружать данные для всех dropdown'ов
- ✗ Требует хорошего UX-дизайна

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — необходимый evil

### 10. 🔴 Sidebar (боковая панель)

**Суть:** Основная навигация: GTD-статусы, проекты, контексты, области, теги с счётчиками.

**Сложность:** ⭐⭐⭐ (3/5)

**Техническая реализация:**
- FE: компонент Sidebar с 5 секциями
- Badge-счётчики (запрос `/api/tasks/counts`)
- Mobile: slide-over, Desktop: collapsible
- Роутинг + фильтрация

**ЗА:**
- ✓ Централизованная навигация — стандартный паттерн
- ✓ Счётчики дают мгновенный обзор
- ✓ Мини-прогрессбары у проектов визуализируют состояние
- ✓ Responsive design (mobile + desktop)

**ПРОТИВ:**
- ✗ Дополнительный endpoint для счётчиков
- ✗ Нужно обновлять счётчики при каждом изменении
- ✗ Mobile sidebar может перекрывать контент
- ✗ Много элементов — может быть перегружен

**Вердикт:** ✅ **ОБЯЗАТЕЛЬНО** — стандартная паттерна, нужна для UX

## Общая оценка

### Все фичи — обязательные

**Почему:**
1. **GTD-приложение** — это ядро продукта, без этих фич нет GTD
2. **Стандартные ожидания** — пользователи ожидают эти фичи в todo-приложении
3. **Синергия** — фичи усиливают друг друга (контексты + проекты + теги = мощная система)
4. **Низкий риск** — все фичи хорошо изучены, есть best practices

### Порядок реализации (рекомендуемый)

1. **Модель данных** — базы всего
2. **Контексты, Области, Теги** — простые CRUD-сущности (можно параллельно)
3. **Проекты** — зависят от областей
4. **GTD-статусы + Inbox** — зависят от расширения модели задач
5. **Подзадачи** — зависят от GTD
6. **Фильтрация и поиск** — зависят от всех сущностей
7. **Sidebar + Форма + Роутинг** — финишная прямая

### Риски

| Риск | Митигация |
|------|-----------|
| Перегрузка UI (слишком много фич) | Progressive disclosure, настройки видимости |
| Сложность фильтрации | Покрыть тестами все комбинации |
| Медленный поиск с LIKE | FTS5 как оптимизация (отдельная задача) |
| N+1 запросы | Eager loading через SQLAlchemy |
| Хаос в тегах | Автодополнение, слияние дубликатов (будущие улучшения) |

## Заключение

**Все 11 фич — обязательны** для полноценного GTD-приложения. Они хорошо продуманы, имеют чёткую цель и логическую взаимосвязь.

**Ключевые преимущества этого набора фич:**
- Полное покрытие GTD-методологии
- Стандартные паттерны (знакомы пользователям)
- Масштабируемость (фичи можно улучшать)
- Низкий риск (проверенные решения)

**Рекомендация:** Реализовать все фичи в указанном порядке.
