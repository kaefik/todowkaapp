  ```
  backend/
  ├── app/

  │   ├── config.py           # Pydantic Settings (env vars)

  └── tests/
  ```

## 2. База данных

### 2.1 Модель `users`
```
users
├── id              UUID, PK
├── username        VARCHAR(50), UNIQUE, NOT NULL, INDEX
├── email           VARCHAR(255), UNIQUE, NOT NULL, INDEX
├── password_hash   VARCHAR(255), NOT NULL
├── is_active       BOOLEAN, DEFAULT true
├── created_at      TIMESTAMP, DEFAULT now()
└── updated_at      TIMESTAMP, DEFAULT now(), ON UPDATE now()
```

### 2.2 Модель `tasks`
```
tasks
├── id              UUID, PK
├── user_id         UUID, FK → users.id, NOT NULL, INDEX
├── title           VARCHAR(255), NOT NULL
├── description     TEXT, NULLABLE
├── is_completed    BOOLEAN, DEFAULT false
├── created_at      TIMESTAMP, DEFAULT now()
└── updated_at      TIMESTAMP, DEFAULT now(), ON UPDATE now()
```

### 2.3 SQL-индексы
- `idx_tasks_user_id` на `tasks.user_id`
- `idx_tasks_user_completed` на `(user_id, is_completed)` — для фильтрации
- `idx_users_username` на `users.username`
- `idx_users_email` на `users.email`

### 2.4 Миграция
- Создать начальную Alembic-миграцию `001_initial_schema`
- Убедиться, что WAL mode включён при инициализации подключения

## 3. Аутентификация

### 3.1 Backend — регистрация
- Endpoint `POST /api/auth/register`
- Принимает: `username`, `email`, `password`
- Валидация: Pydantic schema + zod-подобная проверка (длина пароля ≥ 8, валидный email)
- Хэширование пароля: `bcrypt` (passlib)
- **Guard от открытой регистрации:** env-переменная `REGISTRATION_ENABLED=true/false` (или `INVITE_CODE`). Если отключена — 403
- Возвращает: `user` объект (без password_hash)

### 3.2 Backend — логин
- Endpoint `POST /api/auth/login`
- Принимает: `username` (или email), `password`
- Проверка пароля через bcrypt verify
- Генерация:
  - **Access token** (JWT, short-lived, 15 мин) → возвращается в теле ответа
  - **Refresh token** (JWT, long-lived, 7 дней) → ставится как `HttpOnly`, `Secure`, `SameSite=Strict` cookie
- Возвращает: `{ access_token, token_type, user }`

### 3.3 Backend — refresh
- Endpoint `POST /api/auth/refresh`
- Читает refresh token из HttpOnly cookie
- Валидирует, генерирует новую пару access + refresh
- Ротация refresh token (старый инвалидируется)

### 3.4 Backend — логаут
- Endpoint `POST /api/auth/logout`
- Удаляет refresh cookie (set max_age=0)

### 3.5 Backend — получение текущего пользователя
- Endpoint `GET /api/auth/me`
- Dependency `get_current_user` — извлекает и валидирует access token из заголовка `Authorization: Bearer <token>`
- Возвращает данные текущего пользователя

### 3.6 Frontend — страницы auth
- Страница `/login` — форма логина (react-hook-form + zod validation)
- Страница `/register` — форма регистрации
- Zustand store `useAuthStore`:
  - state: `user`, `accessToken`, `isAuthenticated`, `isLoading`
  - actions: `login()`, `register()`, `logout()`, `refreshToken()`
- Access token хранить **только в памяти** (Zustand store, не localStorage)
- Axios/fetch interceptor:
  - Автоматически прикладывает `Authorization: Bearer` header
  - При 401 — автоматически пытается refresh, при неудаче — redirect на `/login`
- Protected route wrapper: редирект на `/login` если не аутентифицирован

## 4. Задачи — CRUD

### 4.1 Backend API endpoints
| Метод | URL | Описание |
|-------|-----|----------|
| `GET` | `/api/tasks` | Получить все задачи текущего пользователя (с пагинацией `?limit=50&offset=0`) |
| `POST` | `/api/tasks` | Создать задачу |
| `GET` | `/api/tasks/{id}` | Получить задачу по ID |
| `PUT` | `/api/tasks/{id}` | Обновить задачу |
| `PATCH` | `/api/tasks/{id}/toggle` | Переключить `is_completed` |
| `DELETE` | `/api/tasks/{id}` | Удалить задачу |

- Каждый endpoint проверяет, что задача принадлежит текущему пользователю (403 иначе)
- Pydantic schemas:
  - `TaskCreate`: `title` (required), `description` (optional)
  - `TaskUpdate`: `title` (optional), `description` (optional), `is_completed` (optional)
  - `TaskResponse`: все поля + `id`, `created_at`, `updated_at`
  - `TaskListResponse`: `items: list[TaskResponse]`, `total: int`

### 4.2 Backend — сервисный слой
- `TaskService` класс:
  - `get_tasks(user_id, limit, offset)` → list + count
  - `get_task(user_id, task_id)` → task or 404
  - `create_task(user_id, data)` → task
  - `update_task(user_id, task_id, data)` → task
  - `toggle_task(user_id, task_id)` → task
  - `delete_task(user_id, task_id)` → None

### 4.3 Frontend — хук `useTasks()`
- **Абстракция данных** (по дизайн-документу): в итерации 1 хук `useTasks()` работает напрямую с API. В итерации 3 будет переключён на IndexedDB. UI-компоненты не меняются.
- Интерфейс хука:
  ```ts
  useTasks() → {
    tasks: Task[]
    isLoading: boolean
    error: string | null
    addTask(data: CreateTask): Promise<void>
    updateTask(id: string, data: UpdateTask): Promise<void>
    toggleTask(id: string): Promise<void>
    deleteTask(id: string): Promise<void>
    refetch(): Promise<void>
  }
  ```

## 9. Детализация нефункциональных требований (Итерация 1)

### 9.1 Безопасность
| Требование | Реализация | Задача # |
|---|---|---|
| Пароли не хранятся в открытом виде | bcrypt через passlib | 12 |
| Access token short-lived | 15 мин (настраивается через env) | 14 |
| Refresh token в HttpOnly cookie | `Set-Cookie: HttpOnly; Secure; SameSite=Strict` | 14 |
| Access token только в памяти | Zustand store, **не** localStorage | 20 |
| Guard от открытой регистрации | Env `REGISTRATION_ENABLED` | 13 |
| Изоляция данных | Каждый endpoint проверяет `user_id` | 25–30 |
| CORS | Настроить `CORSMiddleware` — только разрешённые origins | 2 |

### 9.2 Производительность
| Требование | Реализация |
|---|---|
| Быстрый старт БД | SQLite + WAL mode |
| Индексы для частых запросов | `idx_tasks_user_completed` — основной запрос списка |
| Async I/O | FastAPI + aiosqlite — неблокирующий доступ к БД |
| Кэширование статики | Nginx: `Cache-Control` для JS/CSS с content hash |

## 11. API контракты (OpenAPI summary)

### Auth endpoints
```
POST   /api/auth/register
  Request:  { username: str, email: str, password: str }
  Response: 201 { id, username, email, created_at }
  Errors:   400 (validation), 403 (registration disabled), 409 (duplicate)

POST   /api/auth/login
  Request:  { username: str, password: str }
  Response: 200 { access_token, token_type, user: {...} }
  Cookie:   Set-Cookie: refresh_token=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/api/auth
  Errors:   401 (invalid credentials)

POST   /api/auth/refresh
  Cookie:   Cookie: refresh_token=<jwt>
  Response: 200 { access_token, token_type }
  Cookie:   Set-Cookie: refresh_token=<new_jwt>; ...
  Errors:   401 (invalid/expired refresh token)

POST   /api/auth/logout
  Response: 200 { message: "Logged out" }
  Cookie:   Set-Cookie: refresh_token=; Max-Age=0

GET    /api/auth/me
  Header:   Authorization: Bearer <access_token>
  Response: 200 { id, username, email, is_active, created_at }
  Errors:   401 (no/invalid token)
```

## 13. Риски итерации 1 и митигации

| Риск | Вероятность | Импакт | Митигация |
|---|---|---|---|
| aiosqlite + WAL блокировки при конкурентной записи | Низкая (1 user) | Средний | WAL mode + retry при `database is locked` |
| Refresh token утечка через XSS | Низкая | Высокий | HttpOnly + Secure + SameSite=Strict cookie |
| Потеря access token при перезагрузке страницы | Средняя | Низкий | Автоматический вызов `/refresh` при загрузке приложения |
| Docker-volume для SQLite потерян | Средняя | Высокий | Чёткая документация volumes в docker-compose |
