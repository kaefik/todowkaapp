# Тестирование офлайн-редактирования задач с персистентным сохранением

**Дата:** 16 апреля 2026
**Статус:** Реализовано и готово к тестированию

## Описание функциональности

Реализована система персистентного сохранения локальных изменений задач в IndexedDB. Теперь если пользователь редактирует задачу офлайн и перезагружает страницу, изменения сохраняются и синхронизируются с сервером при восстановлении соединения.

## Как протестировать

### Сценарий 1: Редактирование задачи офлайн с перезагрузкой страницы

**Шаги:**
1. Запустите фронтенд: `cd frontend && npm run dev`
2. Запустите бэкенд: `cd backend && ./run.sh`
3. Откройте приложение в браузере (http://localhost:5178)
4. Войдите в систему
5. Создайте тестовую задачу (например, "Тестовая задача")
6. Остановите бэкенд (Ctrl+C в терминале бэкенда)
7. Откройте созданную задачу для редактирования
8. Измените описание задачи на "Тестовое описание офлайн"
9. Нажмите "Save"
10. Откройте консоль браузера (F12)
11. Закройте модальное окно
12. **Перезагрузите страницу (F5)**
13. Откройте ту же задачу

**Ожидаемый результат:** описание должно быть "Тестовое описание офлайн"

14. Запустите бэкенд снова: `cd backend && ./run.sh`
15. Дождитесь сообщения "Сеть восстановлена" или "Синхронизация..."
16. Перезагрузите страницу
17. Откройте задачу

**Ожидаемый результат:** описание должно быть "Тестовое описание офлайн" (синхронизировано с сервером)

### Сценарий 2: Использование инструмента отладки

**Показать все локальные изменения:**
```javascript
window.localTaskDebug.getAll()
```
Ожидаемый результат: таблица с локальными изменениями

**Показать изменения для конкретной задачи:**
```javascript
window.localTaskDebug.get('task-id')
```
Замените 'task-id' на ID задачи

**Очистить все локальные изменения:**
```javascript
window.localTaskDebug.clearAll()
```

### Сценарий 3: Несколько изменений офлайн

1. Остановите бэкенд
2. Создайте 3-4 новые задачи
3. Отредактируйте несколько существующих задач
4. Перезагрузите страницу
5. **Ожидаемый результат:** все изменения должны быть видны
6. Запустите бэкенд
7. Дождитесь синхронизации
8. Перезагрузите страницу
9. **Ожидаемый результат:** все изменения должны быть синхронизированы

## Диагностика проблем

### Проблема: Изменения не сохраняются после перезагрузки

**Проверьте в консоли:**
1. ✅ Есть ли логи `[updateTaskMutation]` при сохранении?
2. ✅ Есть ли логи `[localTaskChanges]` при сохранении?
3. ✅ Есть ли логи `Local change saved successfully`?
4. ✅ Есть ли логи `[useTask]` при загрузке задачи?
5. ✅ Есть ли логи `Local change found` при загрузке задачи?

**Используйте инструмент отладки:**
```javascript
window.localTaskDebug.getAll()
```
Должна быть таблица с локальными изменениями.

### Проблема: Индикатор "🔌 Офлайн" не появляется

**Проверьте:**
1. Бэкенд действительно остановлен?
2. Откройте DevTools → Network → "Offline" режим
3. Перезагрузите страницу

### Проблема: Изменения синхронизируются, но потом исчезают

**Проверьте логи:**
1. Есть ли лог `[updateTaskMutation] Update successful`?
2. Есть ли лог `clearing local changes for task`?

**Возможная причина:** Бэкенд не сохраняет изменения. Проверьте логи бэкенда.

```javascript
// Проверить IndexedDB напрямую
indexedDB.open('todowka-local-changes').then(db => {
  const tx = db.transaction('local_task_changes', 'readonly')
  tx.objectStore('local_task_changes').getAll().onsuccess = (e) => {
    console.table(e.target.result)
  }
})

// Очистить все локальные изменения
window.localTaskDebug.clearAll()

// Очистить изменения для конкретной задачи

```

## Архитектура решения

### IndexedDB (todowka-local-changes)
- **Хранилище:** `local_task_changes`
- **Ключ:** `task_id`
- **Структура:**
  ```typescript
  {
    task_id: string
    changes: UpdateTask
    timestamp: number
  }
  ```

### Flow операций

1. **Сохранение (onMutate):**
   ```
   updateTaskMutation.onMutate
   → Оптимистичное обновление React Query
   → Сохранение в IndexedDB
   → Добавление в офлайн-очередь (если ошибка сети)
   ```

2. **Ошибка (onError):**
   ```
   updateTaskMutation.onError
   → Если OfflineQueueError: сохранить изменения
   → Если другая ошибка: сохранить изменения (ВАЖНО!)
   ```

3. **Успех (onSuccess):**
   ```
   updateTaskMutation.onSuccess
   → Удалить из IndexedDB
   → Инвалидировать кеш
   ```

4. **Загрузка (useTask):**
   ```
   useTask
   → Загрузить задачу из API/кеша
   → Загрузить локальные изменения из IndexedDB
   → Слить изменения
   → Вернуть объединенную задачу
   ```

5. **Синхронизация:**
   ```
   useOfflineQueue.syncQueue
   → Обработать все мутации в очереди
   → Если все успешно: очистить IndexedDB
   ```

## Известные ограничения

1. **Редкий кейс:** Если пользователь закрывает браузер сразу после нажатия "Save", но до сохранения в IndexedDB, изменения могут быть потеряны.
   - **Решение:** (не реализовано) Добавить подтверждение перед закрытием

2. **Конфликт изменений:** Если задача была изменена на другом устройстве, локальные изменения могут перезаписать удаленные изменения.
   - **Решение:** (не реализовано) Добавить merge-стратегию с timestamps

3. **Порядок операций:** Мутации синхронизируются в порядке создания (timestamp), но это не гарантирует корректность при конфликтующих изменениях одной задачи.
   - **Решение:** (не реализовано) Добавить детекцию конфликтов
