## КРИТИЧЕСКИЕ ПРОБЛЕМЫ

### 1. Противоречие с timezone

**Проблема:**
```python
reminder_time: time | NULL  # Строка 48 плана
```

**Почему это проблема:**
- Time без даты нельзя корректно конвертировать между timezone
- Если пользователь в timezone, отличной от UTC, напоминание придет не в то время

**Последствие:**
- Напоминания приходят в неправильное время для пользователей в разных timezone

**Решение:**
Заменить на `reminder_datetime: datetime | NULL` или хранить как строку в формате "HH:MM" + использовать user.timezone для конвертации.

### 2. Ambiguity в генерации следующей задачи

**Проблема:**
План не определяет — когда генерировать следующую задачу?

**Варианты:**
- Option A: Сразу после выполнения текущей задачи
- Option B: При наступлении due_date текущей задачи

**Почему это важно:**
- Влияет на UX (пользователь видит следующую задачу сразу или через время)
- Влияет на логику отмены рекурренции
- Влияет на пропущенные выполнения

**Решение:**
Добавить в план четкое правило генерации с обоснованием. Рекомендуется Option A для лучшего UX.

---

### 3. DST (переход на летнее/зимнее время)

**Проблема:**
План не определяет поведение при переходе на летнее/зимнее время.

**Сценарий:**
Задача повторяется "каждый день в 9:00". Происходит переход на летнее время:
- Вариант A: Всегда 9:00 localtime (может быть 8:00 или 10:00 UTC)
- Вариант B: Всегда 9:00 UTC (смещение на 1 час в localtime)

**Почему это важно:**
- В России нет перехода, но приложение может быть интернациональным
- Разные пользователи в разных timezone с разными правилами DST

**Решение:**
Определить стратегию. Рекомендуется: хранить и обрабатывать все datetime в UTC, конвертировать в user.timezone только для отображения и уведомлений. При этом 9:00 localtime всегда 9:00 localtime (включая DST).

### 4. Пропущенные выполнения

**Проблема:**
План не определяет поведение, если пользователь был офлайн несколько дней.

**Сценарий:**
- Задача ежедневная
- Пользователь офлайн 3 дня
- При reconnect: что делать?

**Варианты:**
- Генерировать все 3 пропущенные задачи
- Генерировать только одну — следующую
- Пропустить пропущенные, генерировать только будущие

**Почему это важно:**
- Влияет на состояние inbox/next lists
- Влияет на user experience (перегрузка задачами)
- Влияет на analytics

**Решение:**
Добавить бизнес-правило. Рекомендуется: генерировать только пропущенные задачи за последние N дней (например, 7 дней) и все будущие задачи.

### 5. Race condition в scheduler

**Проблема:**
Если APScheduler запущен на нескольких workers (например, при горизонтальном масштабировании):
- Дублирование задач
- Дублирование уведомлений

**Почему это проблема:**
- In-process scheduler не предназначен для distributed systems
- При рестарте или масштабировании могут появиться дубликаты

**Решение:**
1. Использовать distributed lock (Redis)
2. ИЛИ ограничить до 1 worker (documentation)
3. ИЛИ использовать внешнюю queue (Celery, Redis)

Для текущей реализации: добавить warning в документацию о limitations и использовать flag для single-worker mode.

### 6. Множественные напоминания для одной задачи

**Проблема:**
Как предотвратить повторные напоминания для одной задачи?

**Сценарий:**
- Задача должна быть выполнена сегодня
- Напоминание отправлено в 9:00
- Задача не выполнена
- Следующий запуск scheduler в 9:01 — отправить ли еще раз?

**Почему это проблема:**
- Спам уведомлений
- Annoying для пользователя

**Решение:**
Добавить механизм для отслеживания отправленных напоминаний:
- Поле `last_reminder_sent_at` в Task
- ИЛИ проверка существования notification за последние N часов

### 7. SSE масштабируемость

**Проблема:**
1000 пользователей = 1000 открытых SSE соединений. В in-process APScheduler это проблема.

**Почему это проблема:**
- Ограничение ресурсов сервера
- Potential memory leaks
- Bad user experience при перегрузке

**Решение:**
1. Ограничить количество одновременных соединений
2. ИЛИ использовать Redis pub/sub для distributed SSE
3. ИЛИ использовать внешние сервисы (Pusher, Ably)

Для текущей реализации: добавить connection limit и monitoring.

---

### 8. Offline queue — порядок выполнения

**Проблема:**
Если офлайн создали задачу A, затем изменили задачу B. При reconnect — какой порядок?

**Варианты:**
- A → B (порядок создания)
- B → A (порядок в queue)
- По timestamp

**Почему это важно:**
- Если B зависит от A — порядок критичен
- Last write wins может привести к потере данных

**Решение:**
Сохранять timestamp для каждого mutation в offline queue. Сортировать по timestamp при выполнении.

---

### 9. Конфликт изменений при sync

**Проблема:**
Offline изменили задачу → кто-то другой тоже изменил → sync. Какой вариант победит?

**Варианты:**
- Last write wins
- Merge (конфликтующие поля перезаписываются)
- Manual resolution (показать пользователю)

**Почему это важно:**
- Потеря данных при last write wins
- Сложная UX при manual resolution

**Решение:**
Определить стратегию conflict resolution. Рекомендуется: last write wins с warning в UI о потенциальном конфликте (для v1).

---

### 10. APScheduler persistence

**Проблема:**
При рестарте сервера scheduled jobs теряются:
- Пропущенные задачи?
- Пропущенные напоминания?

**Почему это проблема:**
- Потеря данных
- Несоответствие между планированием и реальностью

**Решение:**
1. Использовать SQLAlchemyJobStore для persistence scheduled jobs
2. ИЛИ rescan при startup и восстановить пропущенное

Рекомендуется: SQLAlchemyJobStore + startup rescan для восстановления пропущенного.

**Проблема:**
API без rate limiting. Может быть DDOS через создание задач/уведомлений.

**Почему это проблема:**
- Security risk
- Resource exhaustion
- Bad user experience для legit users

**Решение:**
Добавить rate limiting (например, slowapi для FastAPI). Можно отложить до production.
