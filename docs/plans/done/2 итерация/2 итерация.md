## 1. Расширение модели данных

### 1.1 Новая модель `contexts`
```
contexts
├── id              UUID, PK
├── user_id         UUID, FK → users.id, NOT NULL, INDEX
├── name            VARCHAR(100), NOT NULL
├── color           VARCHAR(7), NULLABLE (hex цвет, e.g. #FF5733)
├── icon            VARCHAR(50), NULLABLE (emoji или icon name)
├── created_at      TIMESTAMP, DEFAULT now()
└── updated_at      TIMESTAMP, DEFAULT now()

UNIQUE(user_id, name)
```

### 1.2 Новая модель `areas`
```
areas
├── id              UUID, PK
├── user_id         UUID, FK → users.id, NOT NULL, INDEX
├── name            VARCHAR(100), NOT NULL
├── description     TEXT, NULLABLE
├── created_at      TIMESTAMP, DEFAULT now()
└── updated_at      TIMESTAMP, DEFAULT now()

UNIQUE(user_id, name)
```

### 1.3 Новая модель `tags`
```
tags
├── id              UUID, PK
├── user_id         UUID, FK → users.id, NOT NULL, INDEX
├── name            VARCHAR(50), NOT NULL
├── color           VARCHAR(7), NULLABLE
├── created_at      TIMESTAMP, DEFAULT now()

UNIQUE(user_id, name)
```

### 1.4 Связующая таблица `task_tags` (many-to-many)
```
task_tags
├── task_id         UUID, FK → tasks.id, PK
├── tag_id          UUID, FK → tags.id, PK

INDEX(task_id)
INDEX(tag_id)
```

### 1.5 Новая модель `projects`
```
projects
├── id              UUID, PK
├── user_id         UUID, FK → users.id, NOT NULL, INDEX
├── title           VARCHAR(255), NOT NULL
├── description     TEXT, NULLABLE
├── status          VARCHAR(20), DEFAULT 'active'   (active / completed / on_hold)
├── area_id         UUID, FK → areas.id, NULLABLE, INDEX
├── created_at      TIMESTAMP, DEFAULT now()
└── updated_at      TIMESTAMP, DEFAULT now()
```

### 1.6 Расширение модели `tasks`
Добавляемые поля к существующей таблице:
```
tasks (ALTER — новые поля)
├── gtd_status      VARCHAR(20), DEFAULT 'inbox'    (inbox / next_action / waiting / someday / completed)
├── context_id      UUID, FK → contexts.id, NULLABLE, INDEX
├── area_id         UUID, FK → areas.id, NULLABLE, INDEX
├── project_id      UUID, FK → projects.id, NULLABLE, INDEX
├── parent_task_id  UUID, FK → tasks.id, NULLABLE, INDEX  (для подзадач)
├── position        INTEGER, DEFAULT 0               (порядок сортировки внутри списка)
├── due_date        DATE, NULLABLE, INDEX
├── notes           TEXT, NULLABLE
```

### 1.7 Новые SQL-индексы
- `idx_tasks_gtd_status` на `(user_id, gtd_status)` — основной фильтр GTD
- `idx_tasks_context` на `(user_id, context_id)`
- `idx_tasks_project` на `(user_id, project_id)`
- `idx_tasks_area` на `(user_id, area_id)`
- `idx_tasks_parent` на `(user_id, parent_task_id)`
- `idx_tasks_due_date` на `(user_id, due_date)` — для сортировки по дедлайну
- `idx_tasks_position` на `(user_id, gtd_status, position)` — для сортировки внутри списка
- `idx_projects_user_status` на `(user_id, status)`

### 1.8 Миграции Alembic
- `002_add_contexts_areas_tags` — новые таблицы contexts, areas, tags, task_tags
- `003_add_projects` — таблица projects
- `004_extend_tasks_gtd` — новые поля в tasks (gtd_status, context_id, area_id, project_id, parent_task_id, position, due_date, notes)

## 2. Inbox — быстрый захват задач

### 2.1 Backend
- Изменить `POST /api/tasks` — если не передан `gtd_status`, по умолчанию `inbox`
- Endpoint `GET /api/tasks?gtd_status=inbox` — получение задач Inbox
- Endpoint `PATCH /api/tasks/{id}/clarify` — быстрое перемещение из Inbox в другой GTD-статус:
  ```json
  {
    "gtd_status": "next_action",
    "context_id": "uuid",
    "project_id": "uuid",
    "due_date": "2026-05-01"
  }
  ```

### 2.2 Frontend
- **Quick Capture Bar** — всегда видимый/доступный компонент:
  - Поле ввода + Enter = задача мгновенно попадает в Inbox
  - Клавиатурный шорткат (например `N` или `Ctrl+K`) для быстрого фокуса
- Страница `/inbox`:
  - Список задач со статусом `inbox`
  - Для каждой задачи: кнопки быстрого перемещения (→ Next Action, → Waiting, → Someday, → Trash)
  - Inline-форма уточнения: назначить контекст, проект, дедлайн

## 3. GTD-статусы и Workflow

### 3.1 Backend
- Расширить `TaskService`:
  - `get_tasks_by_gtd_status(user_id, status, limit, offset)` → list + count
  - `move_task(user_id, task_id, new_status)` → task
  - `reorder_task(user_id, task_id, new_position)` → task
  - Валидация переходов статусов (опционально — на первом этапе разрешить любой переход)
- Обновить `GET /api/tasks`:
  - Query params: `?gtd_status=inbox&context_id=...&area_id=...&project_id=...&tag_id=...&search=...&sort_by=position&sort_order=asc`
  - Поддержка множественных фильтров одновременно
- Endpoint `PATCH /api/tasks/{id}/move` — смена `gtd_status`
- Endpoint `PATCH /api/tasks/{id}/reorder` — смена `position` (принимает `{ position: int }`)

### 3.2 Pydantic schemas (обновления)
- `TaskCreate` — добавить: `gtd_status?`, `context_id?`, `area_id?`, `project_id?`, `parent_task_id?`, `due_date?`, `notes?`, `tag_ids?: list[UUID]`
- `TaskUpdate` — добавить все те же optional поля
- `TaskResponse` — добавить все новые поля + `tags: list[TagResponse]`, `subtasks_count: int`
- `TaskListResponse` — добавить `filters_applied: dict`

### 3.3 Frontend — GTD-представления
- Боковая панель (sidebar) с навигацией:
  - 📥 **Inbox** (`/inbox`) — входящие
  - ⚡ **Next Actions** (`/next`) — следующие действия
  - ⏳ **Waiting For** (`/waiting`) — ожидание
  - 💭 **Someday / Maybe** (`/someday`) — когда-нибудь
  - 📁 **Projects** (`/projects`) — проекты
  - ✅ **Completed** (`/completed`) — выполненные
- Каждое представление — отфильтрованный список задач с соответствующим `gtd_status`
- Badge/counter задач в каждой категории sidebar

## 9. Фильтрация и поиск

### 9.1 Backend
- Расширить `GET /api/tasks` — комбинированные фильтры:
  ```
  GET /api/tasks?
    gtd_status=next_action&
    context_id=uuid&
    area_id=uuid&
    project_id=uuid&
    tag_id=uuid&
    is_completed=false&
    has_due_date=true&
    due_date_from=2026-04-01&
    due_date_to=2026-04-30&
    search=текст&
    sort_by=position|created_at|due_date|title&
    sort_order=asc|desc&
    limit=50&
    offset=0
  ```
- **Полнотекстовый поиск:** SQLite FTS5 virtual table или `LIKE %query%` (для начала LIKE, FTS5 — как оптимизация)
- Поиск по полям: `title`, `description`, `notes`
- Для SQLite — начальная реализация через `LIKE %query%` (case-insensitive через `COLLATE NOCASE`)
- Оптимизация: при росте данных — переход на FTS5 virtual table (отдельная задача, не блокирующая)
- Сервисный слой `TaskFilterService`:
  - `build_query(user_id, filters: TaskFilterParams)` → SQLAlchemy query
  - `TaskFilterParams` — Pydantic model со всеми фильтрами (все optional)

### 9.2 Frontend
- **Панель фильтров** (над списком задач):
  - Dropdown: GTD-статус (или уже выбран через sidebar)
  - Dropdown: Контекст
  - Dropdown: Область
  - Dropdown: Проект
  - Multi-select: Теги
  - Date range picker: дедлайн от — до
  - Кнопка «Сбросить фильтры»
- **Строка поиска:**
  - Поле ввода с debounce 300ms
  - Иконка поиска + крестик очистки
  - Поиск отправляет `?search=query` к API
  - Подсветка найденного текста в результатах (highlight)
- **Сортировка:**
  - Dropdown: По позиции (по умолчанию), По дате создания, По дедлайну, По названию
  - Toggle: ↑ Ascending / ↓ Descending
- **Сохранение состояния фильтров** в URL query params (`/next?context_id=xxx&sort_by=due_date`) — чтобы можно было делиться ссылкой / обновлять страницу без потери фильтров

## 19. Риски итерации 2 и митигации

| Риск | Вероятность | Импакт | Митигация |
|---|---|---|---|
| Сложность комбинированных фильтров — баги при пересечении | Средняя | Средний | `TaskFilterService.build_query()` покрыт тестами на все комбинации; каждый фильтр добавляется как отдельный `.where()` clause |
| Разрастание формы задачи — UX перегрузка | Средняя | Средний | Прогрессивное раскрытие (progressive disclosure): базовые поля видны, остальные в «Дополнительно» (expandable section) |
| N+1 запросы при загрузке задач с тегами/подзадачами | Высокая | Средний | SQLAlchemy `selectinload` / `joinedload` для eager loading тегов; `subtasks_count` через subquery, а не отдельный запрос |
| Миграции ломают существующие данные из итерации 1 | Низкая | Высокий | Все новые поля — `NULLABLE` или с `DEFAULT`; обратная совместимость; тест миграции на копии production DB |
| Sidebar на мобильном перекрывает контент | Средняя | Низкий | Overlay + backdrop + click-outside-to-close; z-index management |
| Большое количество задач — медленная загрузка | Низкая (1 user) | Низкий | Пагинация по умолчанию (limit=50); индексы на все фильтруемые поля |

## 20. API контракты (новые endpoints)

### Contexts
```
GET    /api/contexts
  Response: 200 [ { id, name, color, icon, created_at, updated_at } ]

POST   /api/contexts
  Request:  { name: str, color?: str, icon?: str }
  Response: 201 { id, name, color, icon, created_at, updated_at }
  Errors:   409 (duplicate name)

PUT    /api/contexts/{id}
  Request:  { name?: str, color?: str, icon?: str }
  Response: 200 { ... }

DELETE /api/contexts/{id}
  Response: 204
  Side-effect: tasks с context_id → NULL
```

### Areas
```
GET    /api/areas
  Response: 200 [ { id, name, description, created_at, updated_at } ]

POST   /api/areas
  Request:  { name: str, description?: str }
  Response: 201 { ... }

PUT    /api/areas/{id}
  Request:  { name?: str, description?: str }
  Response: 200 { ... }

DELETE /api/areas/{id}
  Response: 204
  Side-effect: tasks и projects с area_id → NULL
```

### Tags
```
GET    /api/tags
  Response: 200 [ { id, name, color, created_at } ]

POST   /api/tags
  Request:  { name: str, color?: str }
  Response: 201 { ... }

PUT    /api/tags/{id}
  Request:  { name?: str, color?: str }
  Response: 200 { ... }

DELETE /api/tags/{id}
  Response: 204
  Side-effect: каскадное удаление из task_tags

POST   /api/tasks/{task_id}/tags
  Request:  { tag_ids: [uuid, uuid] }
  Response: 200 { tags: [...] }

DELETE /api/tasks/{task_id}/tags/{tag_id}
  Response: 204
```

### Projects
```
GET    /api/projects?status=active&area_id=uuid
  Response: 200 [ { id, title, description, status, area_id, 
                     tasks_total, tasks_completed, progress_percent,
                     created_at, updated_at } ]

POST   /api/projects
  Request:  { title: str, description?: str, status?: str, area_id?: uuid }
  Response: 201 { ... }

GET    /api/projects/{id}
  Response: 200 { ...project, tasks: [...] }

PUT    /api/projects/{id}
  Request:  { title?: str, description?: str, status?: str, area_id?: uuid }
  Response: 200 { ... }

DELETE /api/projects/{id}
  Response: 204
  Side-effect: tasks с project_id → NULL

GET    /api/projects/{id}/tasks?gtd_status=next_action
  Response: 200 { items: [...], total: int }
```

### Tasks — новые/обновлённые
```
PATCH  /api/tasks/{id}/move
  Request:  { gtd_status: "next_action" | "waiting" | "someday" | "completed" }
  Response: 200 { ...task }

PATCH  /api/tasks/{id}/reorder
  Request:  { position: int }
  Response: 200 { ...task }

PATCH  /api/tasks/{id}/clarify
  Request:  { gtd_status: str, context_id?: uuid, project_id?: uuid, 
              area_id?: uuid, due_date?: date, tag_ids?: [uuid] }
  Response: 200 { ...task }

GET    /api/tasks/{id}/subtasks
  Response: 200 [ { ...subtask } ]

POST   /api/tasks/{id}/subtasks
  Request:  { title: str, description?: str }
  Response: 201 { ...subtask with parent_task_id set }

GET    /api/tasks?gtd_status=inbox&context_id=uuid&area_id=uuid
                 &project_id=uuid&tag_id=uuid&is_completed=false
                 &due_date_from=2026-04-01&due_date_to=2026-04-30
                 &search=текст&sort_by=position&sort_order=asc
                 &limit=50&offset=0
  Response: 200 { items: [...], total: int, filters_applied: {...} }
```
