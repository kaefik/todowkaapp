## Step 2 — Assumption Inversion

### Инверсия 1: Удаление записи + _syncStatus='deleted'

```
Assumption: При deleteTask можно пометить запись _syncStatus='deleted' 
            и позже отправить DELETE на сервер
Inversion:  Если задача физически удаляется из db.tasks 
            (db.tasks.delete(id)), то _syncStatus теряется. 
            Нельзя отправить DELETE на сервер — нет записи.
            Если НЕ удалять (soft-delete), то все useLiveQuery 
            должны фильтровать _syncStatus !== 'deleted'
Impact:     (а) Физическое удаление → потеря мутации, сервер не узнает об удалении
            (б) Soft-delete → все 10+ useLiveQuery вызовов нужно менять, 
            риск забыть фильтр → удалённые задачи появятся в UI
Mitigation: Soft-delete + централизованная функция-фильтр:
            function activeTasks() { return db.tasks.where('_syncStatus').notEqual('deleted') }
            Все хуки используют только activeTasks()
```

### Инверсия 2: Все хуки переписываются за одну итерацию

```
Assumption: Можно удалить React Query в итерации 1b (шаг 1b.5) 
            и оставить справочники на RQ до итерации 2
Inversion:  После удаления QueryClientProvider из main.tsx 
            ВСЕ useQuery/useMutation вызовы сломаются.
            Это: useTasks, useTask, useProjects, useTags, useAreas, useContexts — 
            6 хуков, используемых в 15+ компонентах
Impact:     Приложение полностью неработоспособно между итерациями 1b и 2.
            Справочники (фильтры в TaskFilterPanel, dropdown в TaskEditModal) 
            не загрузятся
Mitigation: Переписать ВСЕ RQ-хуки в итерации 1b (Tasks + Справочники вместе).
            Это увеличит 1b, но гарантирует работоспособность.
            Альтернатива: удалить RQ только в итерации 4 
            (после того как все хуки переписаны)
```

### Инверсия 3: Pull = GET всех данных каждые 5 минут

```
Assumption: GET /api/tasks?limit=100 каждые 5 минут — нормальная нагрузка 
            для персонального GTD-приложения
Inversion:  Нет эндпоинта updated_since. Каждый pull = полный дамп всех данных.
            При 1000 задач — ~100KB JSON каждые 5 минут.
            На мобильном соединении — задержка + трафик
Impact:     (а) Нагрузка на бекенд: N пользователей × 12 запросов/час
            (б) Задержка обновления UI при pull (парсинг + merge)
            (в) Батарея мобильного устройства
Mitigation: Добавить на бекенд параметр updated_since в GET /tasks:
            GET /api/tasks?updated_since=2026-04-18T10:00:00Z
            Возвращает только изменившиеся записи. 
            Это ещё 1-2ч к итерации 0, но значительно оптимизирует sync.
            Если не хочется менять бекенд — увеличить интервал pull до 15 мин,
            а актуальность обеспечивать SSE-событиями
```

## Step 3 — Missing Scenarios

| Сценарий | Риск | Обработка |
|----------|------|-----------|
| **Toggle recurring задачи офлайн** — бекенд при toggle генерирует следующую повторяющуюся задачу. Офлайн этого не происходит | 🟡 | При sync: toggle уйдёт на сервер → сервер сам создаст следующую задачу. Но локально пользователь не увидит новую задачу до sync. Acceptable, но стоит документировать |
| **Move в trash офлайн + Clear trash на сервере** — задача moved в trash локально, но на сервере её уже нет (clear trash выполнили с другого устройства) | 🟢 | Push вернёт 404 → удалить локально. OK |
| **Пользователь быстро создал 5 задач офлайн** — 5 мутаций create в очереди. При push — 5 POST-запросов подряд | 🟢 | OK, но нет batch API. 5 запросов — приемлемо |
| **Initial sync при плохом соединении** — GET /tasks?limit=100 вернул partial JSON (connection dropped mid-response) | 🟡 | fetch выбросит TypeError. SyncEngine должен обработать и повторить. Не описано в error handling |
| **Бекенд добавил новые поля в TaskResponse** (миграция) — клиентский DbTask не содержит новых полей | 🟢 | Dexie не валидирует схему — новые поля просто проигнорируются при pull. При необходимости — Dexie version upgrade |
| **DbMutation.payload для toggle** — toggle не имеет body (`PATCH /tasks/{id}/toggle`), но payload обязателен в схеме DbMutation | 🟢 | payload может быть `null`/пустой строкой для toggle и move. Уточнить в схеме: `payload: string | null` |
| **Update задачи с tag_ids** — текущий API принимает `tag_ids: string[]` в PUT. При sync отправится tag_ids, но DbTask хранит `tagIds`. Маппинг при push: `tagIds → tag_ids` | 🟢 | Описать в mappers.ts: DbTask.tagIds → API body tag_ids. OK, но стоит явно упомянуть |

### Блокеры для разрешения до начала:

**Блокер 1: Прямое использование useQueryClient в компонентах**
Компоненты `NotificationProvider.tsx` и `Trash.tsx` используют `useQueryClient().invalidateQueries()` напрямую. При удалении RQ (шаг 1b.5/1b.6) они сломаются. Нужно добавить шаги для их адаптации.

**Блокер 2: Стратегия удаления (soft-delete vs hard-delete)**
Нужно принять решение: физически удалять задачи из Dexie (и хранить pending deletes отдельно) или использовать soft-delete (`_syncStatus='deleted'`) с фильтрацией во всех запросах.

### Рекомендуемые решения:

1. **Удаление RQ перенести в итерацию 4** — переписать все хуки на Dexie (итерации 1b + 2), но RQ оставить работающим параллельно. Удалить в итерации 4 после того как всё работает. Это безопаснее.

2. **Soft-delete** — `_syncStatus='deleted'`, все `useLiveQuery` вызовы фильтруют `.filter(t => t._syncStatus !== 'deleted')`. При push — отправить DELETE, при успехе — физически удалить.

3. **NotificationProvider** — заменить `queryClient.invalidateQueries({ queryKey: taskKeys.all })` на вызов `SyncEngine.triggerPull()` или Dexie-триггер.

4. **Trash.tsx** — заменить `queryClient.invalidateQueries()` на Dexie-операции.
