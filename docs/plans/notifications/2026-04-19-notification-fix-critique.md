## Step 2 — Инверсия допущений

### Инверсия 1: Множественные оффсеты

```
Допущение:   Добавление sent_reminder_offsets решает проблему множественных оффсетов
Инверсия:    sent_reminder_offsets не решает проблему, потому что find_due_tasks() 
             всё ещё обрабатывает каждую задачу один раз за tick, а с разными оффсетами 
             могут быть разные timing-окна
Влияние:     Оффсет [1440] (1 день до) может быть пропущен, если scheduler tick 
             не попадает в нужную минуту (например, сервер был нагружен)
Устранение:  Хранить не "отправленные оффсеты", а "время следующего напоминания" 
             (precomputed). Scheduler проверяет только now >= next_reminder_at
```

### Инверсия 2: Polling решает проблему SSE

```
Допущение:   Polling каждые 30-60 сек надёжно доставляет уведомления при падении SSE
Инверсия:    Polling создаёт нагрузку на SQLite, которая вызывает даунтаймы, 
             которые ломают и polling, и SSE, и scheduler
Влияние:     Каскадный отказ. 1000 пользователей × 1 req/min = 16 req/sec к БД. 
             SQLite при конкурентных writes уже напрягается с scheduler'ом
Устранение:  Polling только для SSE fallback с exponential backoff: 
             30s → 60s → 120s (cap). + Агрессивное кэширование ответа /api/notifications
```

### Инверсия 3: Recovery за 24 часа достаточно

```
Допущение:   Окно recovery 24 часа покрывает绝大多数 случаев
Инверсия:    Сервер может быть отключён 2-7 дней (хостинг, отпук, переезд)
Влияние:     Все напоминания за период даунтайма потеряны навсегда
Устранение:  Recovery окно = max(7 дней, время даунтайма). Искать все задачи 
             с due_date в пределах [last_known_uptime, now + max_offset], 
             где last_known_uptime записывается в файл при shutdown
```

## Step 3 — Упущенные сценарии

| # | Сценарий | Риск | Как обработать |
|---|----------|------|----------------|
| M-1 | Пользователь меняет timezone в настройках | 🟡 | Существующие reminder_time для задач не пересчитываются. Напоминание, установленное как "09:00", останется 09:00 в НОВОЙ timezone — сдвинется на часов. План не учитывает |
| M-2 | Scheduler tick занимает > 60 сек | 🔴 | При 10K задач find_due_tasks + send_reminder может занять > 60 сек. APScheduler запустит следующий tick параллельно → дублирование уведомлений. План не предусматривает lock/mutex |
| M-3 | Пользователь обновляет задачу во время scheduler tick | 🟡 | Race condition: scheduler отправил напоминание и коммитит → в этот момент пользователь обновляет reminder_time → `reminder_fired` сбрасывается в False → на следующем tick напоминание отправится повторно |
| M-4 | Task удалён (trash → cleanup) между find и send | 🟢 | Scheduler нашёл задачу в find_due_tasks, но она была удалена за время обработки. send_reminder упадёт. Уже обрабатывается try/except, но стоит упомянуть |
| M-5 | Мобильный браузер убивает SSE в фоне | 🟡 | Chrome на Android может убить SSE через 5 минут в фоне. План не учитывает мобильную специфику — polling нужен более агрессивный при visibilitychange |
| M-6 | Два таба одного пользователя | 🟢 | Оба таба подключатся к SSE (лимит 3). Каждый получит уведомление → оба вызовут browser notification → 2 уведомления. Можно решить через `tag` в Notification API (уже частично решено) |
| M-7 | DST переход (переход на летнее/зимнее время) | 🟡 | reminder_time=09:00. DST переход сдвигает UTC offset. В день перехода напоминание может прийти на час раньше или позже. План не рассматривает |

## Verdict

```
VERDICT: 🔴 NEEDS REVISION
```

### Причины:

1. **4 блокера** требуют решения до начала реализации
2. БАГ-1 — центральный баг плана, но миграционная стратегия не проработана (существующие данные, взаимодействие старых и новых полей)
3. БАГ-8, вероятно, не является багом в описанном виде — нужен пересмотр
4. Отсутствие стратегии тестирования при работе с timezone-логикой — прямой путь к регрессии
5. Web Push (БАГ-9) разувеличивает план, превращая багфикс в разработку новой фичи

### Рекомендации по доработке:

1. **Добавить раздел "Миграция данных"** — что происходит с существующими данными при добавлении `sent_reminder_offsets`
2. **Добавить раздел "Тестирование"** — какие тесты обновить/добавить
3. **Добавить `max_instances=1`** для scheduler jobs — предотвращает параллельные tick'и
4. **Пересмотреть БАГ-8** — уточнить реальную проблему (UX vs логика)
5. **Вынести БАГ-9 (Web Push), БАГ-11 (звук), БАГ-12 (пагинация)** в отдельные планы
6. **Явно описать** как `last_reminder_sent_at` и `sent_reminder_offsets` сосуществуют
7. **Убрать `delivered_late`** — over-engineering для recovery
8. **Добавить** handling для M-5 (мобильные браузеры) и M-2 (scheduler timeout)
