**Дата:** 8 апреля 2026

## Критические несоответствия

### 1. GET /api/auth/me - Получение текущего пользователя

**Статус:** ✗ НЕ РЕАЛИЗОВАНО

**Где в бэкенде:** `backend/app/api/auth.py:151-153`

```python
@auth_router.get("/me", response_model=UserResponse)
async def me(current_user: Annotated[User, Depends(get_current_user)]) -> User:
    return current_user
```

**Что возвращает:**
- `id: UUID` - ID пользователя
- `username: str` - Имя пользователя
- `email: EmailStr` - Email пользователя
- `is_active: bool` - Активен ли пользователь
- `created_at: datetime` - Дата регистрации

**Как используется во фронтенде:** Никак

**Проблема:**
- После логина пользователь сохраняется в store только из ответа `/api/auth/login`
- При обновлении страницы информация о пользователе может быть устаревшей или отсутствовать
- Нет возможности синхронизировать состояние пользователя с сервером

**Зачем нужно:**
- Синхронизация состояния пользователя после обновления страницы
- Получение актуальных данных о пользователе
- Проверка статуса аккаунта (is_active)
- Отображение корректной информации в UI

**Где использовать:**
- В `authStore.ts` добавить функцию `fetchCurrentUser()`
- Вызывать при инициализации приложения (если есть токен)
- Вызывать после обновления токена
- Вызывать периодически для синхронизации

**Влияние на UX:**
- Среднее: Пользователь видит устаревшую или некорректную информацию о себе
- Безопасность: Не проверяется актуальный статус is_active

### 2. GET /api/tasks/{id} - Получение конкретной задачи

**Статус:** ✗ НЕ РЕАЛИЗОВАНО

**Где в бэкенде:** `backend/app/api/tasks.py:40-55`

```python
@tasks_router.get("/{task_id}", response_model=TaskResponse)
async def get_task(
    task_id: str,
    current_user: Annotated[User, Depends(get_current_user)],
    db: Annotated[AsyncSession, Depends(get_db)],
) -> TaskResponse:
    service = TaskService(db)
    task = await service.get_task(user_id=current_user.id, task_id=task_id)
    if task is None:
        raise HTTPException(status_code=404, detail="Task not found")
    return task
```

**Что возвращает:** Полные данные задачи (title, description, is_completed, created_at, updated_at)

**Как используется во фронтенде:** Никак

**Текущая реализация:** Задачи загружаются только через `GET /api/tasks` с пагинацией

**Проблема:**
- Если задача была создана или изменена другим клиентом, она не появится в списке до полного обновления
- Нет возможности открыть детальный просмотр задачи с полными данными
- При редактировании используется данные из локального состояния, которые могут быть устаревшими

**Зачем нужно:**
- Детальный просмотр отдельной задачи
- Синхронизация конкретной задачи с сервером перед редактированием
- Проверка существования задачи перед операциями
- Возможность прямого доступа к задаче по ID (deep linking)

**Где использовать:**
- При открытии модального окна редактирования - загрузить актуальные данные
- При реализации страницы детального просмотра задачи
- Для deep linking (открывать задачу по URL /tasks/{id})

**Влияние на UX:**
- Среднее: При редактировании могут быть потеряны изменения, сделанные на другом устройстве
- Низкое: Отсутствует возможность посмотреть детали задачи отдельно

### 3. PATCH /api/tasks/{id}/toggle - Переключение статуса

**Статус:** ✗ НЕ ИСПОЛЬЗУЕТСЯ (есть альтернатива)

**Где в бэкенде:** `backend/app/api/tasks.py:77-92`

```python
@tasks_router.patch("/{task_id}/toggle", response_model=TaskResponse)
async def toggle_task(
    task_id: str,
    current_user: Annotated[User, Depends(get_current_user)],
    db: Annotated[AsyncSession, Depends(get_db)],
) -> TaskResponse:
    service = TaskService(db)
    task = await service.toggle_task(user_id=current_user.id, task_id=task_id)
    if task is None:
        raise HTTPException(status_code=404, detail="Task not found")
    return task
```

**Как реализовано во фронтенде:** Используется `PUT /api/tasks/{id}` с передачей `is_completed`

**Где:** `frontend/src/hooks/useTasks.ts:116-124`

```typescript
const toggleTask = useCallback(async (id: string) => {
  const task = tasks.find((t) => t.id === id)
  if (!task) return

  try {
    await updateTask(id, { completed: !task.completed })
  } catch (err) {
  }
}, [tasks, updateTask])
```

**Проблема:**
- PUT обновляет все поля задачи (title, description, is_completed)
- PATCH toggle более семантически корректен для операции переключения
- PUT отправляет больше данных (title, description), хотя они не меняются
- PATCH toggle может быть оптимизирован на бэкенде для быстрого переключения

**Зачем нужно:**
- Более семантически правильный API для операции переключения
- Оптимизация трафика (отправляется меньше данных)
- Возможность реализации специфической логики для toggle на бэкенде

**Где использовать:**
- Заменить вызов `updateTask` на вызов специализированного `toggleTask` API
- В `useTasks.ts` добавить отдельную функцию для toggle

**Влияние на UX:**
- Низкое: Функционально работает одинаково
- Производительность: Небольшое улучшение скорости и трафика

### 4. registration_enabled - Отключение регистрации

**Статус:** ⚠️ ЧАСТИЧНО РЕАЛИЗОВАНО (на бэкенде)

**Где в бэкенде:** `backend/app/api/auth.py:32-36`

```python
if not settings.registration_enabled:
    raise HTTPException(
        status_code=status.HTTP_403_FORBIDDEN,
        detail="Registration is disabled",
    )
```

**Где в конфиге:** `backend/app/config.py:9`

```python
registration_enabled: bool = True
```

**Как используется во фронтенде:** Не используется

**Проблема:**
- Если registration_disabled = True, пользователь видит форму регистрации и получает ошибку 403
- Нет UI-индикации о том, что регистрация отключена
- Нет возможности скрыть ссылку на регистрацию в форме входа

**Зачем нужно:**
- Администратор может отключить регистрацию новых пользователей
- Предотвращение спама при публичном доступе
- Режим "только по приглашению"

**Где использовать:**
- Добавить GET /api/config для получения настроек приложения
- На основе settings.registration_enabled скрывать кнопку "Sign up" и ссылку
- Показывать сообщение о том, что регистрация отключена

**Влияние на UX:**
- Среднее: Пользователь тратит время на заполнение формы, а потом получает ошибку
- Безопасность: Невозможно полностью контролировать доступ к регистрации

## Заключение

Текущее состояние приложения показывает хороший базовый функционал, но есть существенные упущения:

**Сильные стороны:**
- Базовый CRUD для задач работает
- Аутентификация полностью функциональна
- PWA поддерживается
- Код чистый и структурированный

**Слабые стороны:**
- Нет синхронизации состояния пользователя после обновления страницы
- Не используются все API endpoints
- Много данных не отображается (временные метки, статусы)
- Отсутствуют важные страницы (профиль, настройки)
- Конфигурационные возможности (registration_enabled, invite_code) не интегрированы в UI

**Рекомендация:**
Начать с реализации Priority 1 (GET /api/auth/me и страница /profile), так как это критически влияет на безопасность и пользовательский опыт. Остальное можно реализовывать постепенно в следующих итерациях.
