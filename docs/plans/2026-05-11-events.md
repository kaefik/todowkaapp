# План исправления: Повторяющиеся события календаря

**Дата:** 2026-05-11
**Статус:** К реализации (ревизия v2 после критики)

## Описание проблемы

Три бага в повторяющихся событиях календаря:

### Баг 1: Сдвиг времени следующего события на +3 часа
Событие "тест повтор события ежедневная" с началом 11 мая 21:03–21:04, повтор ежедневный. Следующее событие создано на 12 мая **0:03–0:04** вместо **21:03–21:04**.

### Баг 2: Завершённое событие не отображается
Прошедшее событие 11 мая не видно в календаре. Ожидается: событие остаётся видимым с визуальным маркером "завершено/прошло".

### Баг 3: Sync Engine теряет поля при синхронизации
При записи событий с сервера в Dexie теряются поля `recurrenceType`, `recurrenceConfig`, `recurrenceEndDate`, `location`, `attendees`. Сгенерированные scheduler'ом события теряют данные повтора и место/участников.

---

## Корневые причины

### Баг 1: Двойная конвертация timezone

**Цепочка данных:**

1. Фронтенд (`EventEditorModal.tsx:145-154`, `toIsoDateTime`) формирует `"2026-05-11T21:03:00"` — **без** timezone-offset. Это локальное время пользователя UTC+3.

2. Pydantic парсит как **naive** datetime (без tzinfo).

3. SQLAlchemy + SQLite: колонка `DateTime(timezone=True)` хранит текст `"2026-05-11 21:03:00"`. При чтении SQLAlchemy возвращает **aware** `2026-05-11T21:03:00+00:00` (добавляет UTC).

4. Scheduler (`_job_generate_recurring_events`) вызывает `calculate_next_start_time`:
   - `_to_naive_utc(2026-05-11T21:03:00+00:00)` → `2026-05-11 21:03:00` (убрал +00:00, но это "21:03 UTC")
   - `+1 день` → `2026-05-12 21:03:00` (naive)

5. `generate_next_event` (строка 62):
   - `_to_aware_utc(2026-05-12 21:03:00)` → `2026-05-12T21:03:00+00:00` (UTC)

6. Фронтенд получает `2026-05-12T21:03:00Z`, показывает в UTC+3 → **12 мая 00:03**

**Суть:** Локальное 21:03 (UTC+3 = 18:03 UTC) сохранено как 21:03 UTC. При генерации следующего события сдвиг сохраняется, а при отображении на фронтенде UTC конвертируется обратно в UTC+3 → +3 часа.

**Дополнительная проблема:** `toIsoDateTime` использует `new Date(value).getTimezoneOffset()` — это offset **браузера**, а не timezone из настроек приложения. Если timezone браузера не совпадает с настройками — сдвиг будет другим.

**Затронутые файлы:**
- `frontend/src/components/calendar/EventEditorModal.tsx` — `toIsoDateTime()` не добавляет offset, использует timezone браузера
- `frontend/src/hooks/useCalendarEvents.ts` — `updateEvent` получает `start_time` уже через `toIsoDateTime`, но текущая версия не добавляет offset. После фикса утилиты — исправится автоматически
- `backend/app/services/event_recurrence_service.py` — `catch_up_missed_events` использует `datetime.now()` (naive) в строках 281-282, 289, 294
- `backend/app/scheduler.py` — `_job_generate_recurring_events`, `_job_startup_recovery_events`

### Баг 2: Нет визуального статуса "прошло" для событий

В модели `CalendarEvent` **нет поля `is_completed`**. Событие просто "проходит" по времени. Фронтенд применяет `opacity-50` к прошедшим событиям (`CalendarEventCard.tsx:35`), но:

- В DayView прошедшие события текущего дня отображаются, но визуально почти незаметны (только прозрачность)
- Нет явного маркера "завершено" (зачёркивание, иконка, бейдж)

**Затронутые файлы:**
- `frontend/src/components/calendar/CalendarEventCard.tsx` — только opacity-50

### Баг 3: Sync Engine теряет поля

`syncEngine.ts:158-171` — transform для calendar events не содержит полей `recurrenceType`, `recurrenceConfig`, `recurrenceEndDate`, `location`, `attendees`. При sync с сервера эти данные не записываются в Dexie.

**Затронутые файлы:**
- `frontend/src/db/syncEngine.ts` — transform для calendarEvents

### Шаг 2: Фронтенд — утилита timezone и отправка времени с offset

**Файлы:** `frontend/src/utils/timezone.ts` (новый), `frontend/src/components/calendar/EventEditorModal.tsx`

**Приоритет:** 🔴 Критический

Создать утилитный файл `frontend/src/utils/timezone.ts` и использовать его в `EventEditorModal.tsx` вместо локальной `toIsoDateTime`.

**Ключевые требования:**
- `getTimezoneOffsetStr` вычисляет offset на **дату `start_time`**, а не на `new Date()` — для корректной работы с DST
- Используется `user.timezone` из `useAuthStore`, fallback на `Intl.DateTimeFormat`
- Для all-day событий offset НЕ добавляется
- Строковая конкатенация вместо `new Date()` — нет проблемы Safari

**Новый файл:** `frontend/src/utils/timezone.ts`
```ts
import { useAuthStore } from '../stores/authStore'

export function getUserTimezone(): string {
  const user = useAuthStore.getState().user
  return user?.timezone || Intl.DateTimeFormat().resolvedOptions().timeZone
}

export function getTimezoneOffsetStr(timezone?: string, referenceDate?: Date): string {
  const tz = timezone || getUserTimezone()
  const ref = referenceDate || new Date()
  const utcStr = ref.toLocaleString('en-US', { timeZone: 'UTC' })
  const tzStr = ref.toLocaleString('en-US', { timeZone: tz })
  const offsetMs = new Date(tzStr).getTime() - new Date(utcStr).getTime()
  const offsetMin = offsetMs / (60 * 1000)
  const sign = offsetMin >= 0 ? '+' : '-'
  const absOffset = Math.abs(offsetMin)
  const h = String(Math.floor(absOffset / 60)).padStart(2, '0')
  const m = String(absOffset % 60).padStart(2, '0')
  return `${sign}${h}:${m}`
}

export function toIsoDateTime(value: string | null, isAllDay: boolean): string | null {
  if (!value) return null
  if (isAllDay) {
    return value + 'T00:00:00'
  }
  if (value.includes('T')) {
    const refDate = new Date(value)
    const offset = getTimezoneOffsetStr(undefined, refDate)
    return value + ':00' + offset
  }
  return value + 'T00:00'
}
```

**Изменить** `EventEditorModal.tsx`: удалить локальную `toIsoDateTime` (строки 145-154), импортировать из `utils/timezone.ts`.

**Результат:** Фронтенд отправит `"2026-05-11T21:03:00+03:00"` вместо `"2026-05-11T21:03:00"`. Offset вычисляется на дату события, а не на текущий момент — корректно для DST.

**Примечание:** `useCalendarEvents.ts:updateEvent` получает `start_time` уже через `toIsoDateTime` из `EventEditorModal.tsx:169-170`. После замены локальной функции на утилиту — обновления событий тоже получат offset автоматически. Дополнительных изменений в `useCalendarEvents.ts` не требуется.

## Риски

### 1. Существующие события с неверным timezone
Уже сохранённые события (без offset) продолжат генерироваться со сдвигом.

**Стратегия:** Не делать миграцию (слишком сложно, нужны timezone каждого пользователя). После обновления через UI событие сохранится с правильным offset. Scheduler обрабатывает как naive так и aware datetime корректно (благодаря `_to_naive_utc`).

### 2. All-day события
All-day события НЕ получают timezone-offset — остаются `"2026-05-11T00:00:00"`. Это корректно, т.к. all-day события не привязаны к конкретному времени.

### 3. Offline-создание событий
Если пользователь создаёт событие offline, `useAuthStore.getState().user?.timezone` содержит последнее известное значение. Fallback на timezone браузера через `Intl.DateTimeFormat`. Риск минимален — timezone редко меняется.

### 4. DST (переход на летнее/зимнее время)
`getTimezoneOffsetStr` вычисляет offset на дату `start_time` (аргумент `referenceDate`), а не на текущий момент. Это частично решает проблему: событие созданное на декабрь получит зимний offset. Однако scheduler прибавляет 24 часа к UTC-времени — если повтор попадает на другой сезон, локальное время может сдвинуться на 1 час.

**Фундаментальное ограничение:** scheduler не знает timezone пользователя и не может корректировать DST. Для Europe/Moscow (нет DST с 2014) — не актуально. Для DST-зон — потребуется отдельное улучшение (хранить timezone в событии и использовать библиотеку вроде `pytz`/`zoneinfo`).

### 5. Rollback-стратегия
Если после выкатки timezone-fix ломает отображение:
- Revert коммита с шагом 2 (утилита timezone)
- Очистить Dexie через logout/login (автоматический `clearLocalData`)
- Серверные данные не затрагиваются — `_to_naive_utc` обрабатывает naive datetime корректно

### 6. Sync Engine mergeRecord
При merge локальной записи с серверной — если серверная запись не имеет recurrence-полей (старые данные), а локальная имеет (offline-создание) — `mergeRecord` выбирает по `updatedAt`. Риск потери минимален, но не нулевой. После фикса шага 1 сервер будет всегда возвращать полные данные.
