# Анализ и план исправления оффлайн-функциональности (v2)

**Дата:** 2026-04-16
**Статус:** Требует реализации
**Версия:** 2 (после критического разбора)

---

## 1. Текущее состояние

Инфраструктура оффлайн частично реализована:

- PWA + Service Worker с прекэшем и runtime-caching
- IndexedDB-кэш HTTP-ответов (`todowka-cache`)
- Персистентность React Query через IndexedDB (`todowka-query-cache`)
- Офлайн-очередь мутаций (`todowka-offline`, useOfflineQueue)
- Локальные изменения задач (`todowka-local-changes`, localTaskChanges)
- Optimistic updates **только** для `updateTask`
- OfflineBanner + тосты (дублированные)

**Но всё это не работает из-за критических багов.**

## 2. Найденные баги

### КРИТИЧЕСКИЕ (блокируют работу)

#### БАГ-1: `queueMutationFn` = `null` при старте приложения

**Файл:** `httpClient.ts:45`, `useOfflineQueue.ts`

`useOfflineQueue` монтируется внутри `TaskEditModal` и `SyncIndicator` (который нигде не примонтирован — см. БАГ-8). До первого открытия модалки глобальная функция очереди = `null`. Все мутации молча теряются в офлайне — `httpClient.ts:158` и `httpClient.ts:215` проверяют `if (queueMutationFn && ...)` и получают `false`.

**Исправление:** Примонтировать `useOfflineQueue` на уровне корневого компонента (App/layout), чтобы `queueMutationFn` инициализировался при старте приложения.

---

#### БАГ-2: `addTask` — нет offline-обработки

**Файл:** `useTasks.ts:206-215`

Мутация `addTaskMutation` не имеет:
- `onMutate` — нет optimistic update, задача не появляется в UI
- `onError` с проверкой `OfflineQueueError` — ошибка просачивается как generic

Обёртка `addTask`:
```typescript
catch (err) {
  if (err instanceof ApiError) { throw err }
  throw new Error('Failed to add task')  // OfflineQueueError попадает сюда
}
```

`OfflineQueueError extends Error`, не `ApiError`. Пользователь видит "Failed to add task". Задача не появляется в UI.

**Исправление:**
1. Добавить `onMutate` с optimistic update (создать временную задачу с локальным UUID с префиксом `local_`)
2. Добавить `onError` — при `OfflineQueueError` не откатывать изменения
3. В обёртке `addTask` — подавлять `OfflineQueueError` и возвращать локальную задачу
4. **После sync:** обновить локальный UUID на серверный во всех кэшах (React Query, localTaskChanges) — см. Задачу 1.6

---

#### БАГ-3: `deleteTask` / `toggleTask` / `moveTask` — та же проблема

**Файл:** `useTasks.ts:302-390`

- Нет `onMutate` — визуально ничего не меняется
- Нет `onError` для `OfflineQueueError`
- Обёртки ловят `ApiError`, но `OfflineQueueError` просачивается как generic error

**Исправление:** Добавить optimistic updates + `onError` для `OfflineQueueError` во все мутации.

---

#### БАГ-4: `getAllLocalTaskChanges` не экспортируется

**Файл:** `useLocalTaskChanges.ts:6`, `localTaskChanges.ts`

Импорт функции, которой нет в модуле. Compile-time ошибка.

**Исправление:** Добавить экспорт `getAllLocalTaskChanges` в `localTaskChanges.ts`:
```typescript
export async function getAllLocalTaskChanges(): Promise<Record<string, LocalTaskChange>> {
  const db = await openDB('todowka-local-changes', 1, ...)
  return await db.getAll('task-changes')
}
```

---

#### БАГ-5: HTTP 500+ = "офлайн" (дублирование данных)

**Файл:** `httpClient.ts:141`

```typescript
const isNetworkError = response.status >= 500 || response.status === 0
```

`status >= 500` — сервер ответил (сеть работает), но мутация ставится в очередь. При "восстановлении" сети мутация повторится → дублирование.

**Примечание:** `status === 0` может возникнуть при CORS-блокировке, но не при офлайне (fetch выбрасывает TypeError). Проверка `status === 0` для офлайна — мёртвый код, но может быть полезна для CORS-сценариев.

**Исправление:** Убрать `status >= 500` из условия. Оставить `status === 0`. Ставить в очередь только при `TypeError` от fetch (нет сети) и `status === 0`.

---

#### БАГ-6: `ONLINE_RECONNECT` — никто не слушает

**Файл:** `httpClient.ts:298`

Событие отправляется, но нигде нет `addEventListener('ONLINE_RECONNECT', ...)`. Синхронизация при восстановлении сети может не запуститься (хотя useOfflineQueue имеет свой `online` listener → syncQueue).

**Исправление:** Добавить слушатель `ONLINE_RECONNECT` в `useOfflineQueue`:
```typescript
useEffect(() => {
  const handleReconnect = () => { syncQueue() }
  window.addEventListener('ONLINE_RECONNECT', handleReconnect)
  return () => window.removeEventListener('ONLINE_RECONNECT', handleReconnect)
}, [])
```

---

### СЕРЬЁЗНЫЕ (нарушают UX)

#### БАГ-7: Areas / Projects / Contexts / Tags — без offline

**Файлы:** `useAreas.ts`, `useProjects.ts`, `useContexts.ts`, `useTags.ts`

Мутации этих сущностей не обрабатываются офлайн. Нет optimistic updates, нет `OfflineQueueError` обработки. При офлайне пользователь видит generic ошибку, изменение теряется.

**Исправление:** Создать shared utility для optimistic updates (см. Задачу 3.2) и применить ко всем сущностям.

---

#### БАГ-8: `SyncIndicator` нигде не смонтирован

**Файл:** `SyncIndicator.tsx`

Компонент готов (импортирует `useOfflineQueue`), но не подключён. Пользователь не видит:
- Сколько мутаций в очереди
- Идёт ли синхронизация
- Что данные не синхронизированы

**Исправление:** Примонтировать `SyncIndicator` в layout (в Header или Sidebar).

---

#### БАГ-9: `clearAllLocalTaskChanges` после sync — потеря новых изменений

**Файл:** `useOfflineQueue.ts:213-216`

Если пользователь внёс изменения во время синхронизации, новые локальные изменения будут удалены.

**Исправление:** Заменить `clearAllLocalTaskChanges()` на granular-очистку — удалять только те `localTaskChanges`, чьи мутации успешно синхронизированы. Использовать `entityId` из объекта Mutation (см. Задачу 1.7), а не парсинг URL:
```typescript
for (const mutation of syncedMutations) {
  if (mutation.entityId) {
    await deleteLocalTaskChange(mutation.entityId)
  }
}
```

---

#### БАГ-10: Дублирование toast-уведомлений

**Файл:** `httpClient.ts:284-313`

**Уточнённая диагностика:** Дублирование происходит внутри самого `httpClient.ts`, а НЕ между `httpClient` и `useOfflineQueue` (useOfflineQueue вообще не вызывает тосты). Сценарий:
1. Браузерный `offline` event → toast "Вы офлайн" (строка 305)
2. fetch выбрасывает TypeError → toast "Офлайн режим" (строка 230)

Два тоста для одного события.

**Исправление:** Реорганизовать тосты внутри `httpClient.ts`:
- Убрать toast из `online`/`offline` event listeners
- Оставить тосты только при постановке в очередь (mutation queued) и при восстановлении сети после sync
- Либо: перенести все офлайн-тосты в `useOfflineQueue` (потребует добавить `useToastStore`)

### Этап 2: Синхронизация и compaction (P1) — 4-6 часов

**Задача 2.1: Реализовать compaction очереди мутаций** (2-3 часа)
- Перед sync: анализировать очередь на предмет схлопываемых мутаций
- Правила compaction для одной задачи (по `entityId`):
  - `create` + `update` → `create` с итоговыми данными
  - `create` + `delete` → удалить обе мутации из очереди (noop)
  - `update` + `update` → `update` с итоговыми данными
  - `update` + `delete` → `delete` (предыдущие update не нужны)
- **Проверка:** 5 мутаций одной задачи → после compaction 1-2 мутации

**Задача 2.2: Обработка refresh token при sync** (1-2 часа)
- Перед replay мутаций: проверить валидность access token
- Если истёк — попытаться refresh через `/api/auth/refresh`
- Если refresh тоже падает — остановить sync, показать диалог ре-логина
- Если refresh_token истёк (>7 дней офлайн) — предупредить о потере очереди, очистить
- **Проверка:** Имитация истёкшего токена → sync проходит после refresh

**Задача 2.3: Granular-очистка localTaskChanges** (1 час)
- Файл: `hooks/useOfflineQueue.ts`

## 6. Риски

1. **IndexedDB может быть очищен браузером** при нехватке памяти — Task 3.5 добавляет fallback
2. **Офлайн-создание задач** требует генерации локального UUID и маппинга на серверный ID — Task 1.6
3. **Множественные мутации одной задачи** — схлопываются через compaction — Task 2.1
4. **Параллельное редактирование** на разных устройствах — без версионности возможна потеря данных (Phase 2)
5. **Истёкший токен при sync** — Task 2.2 добавляет refresh token handling
6. **Safari Private Mode** блокирует IndexedDB — Task 3.5 добавляет fallback
7. **Два таба одновременно** — race condition на общую IndexedDB очередь. Митигация: транзакции IndexedDB + флаг `isSyncing`

## 7. Тестовая стратегия

### Ручное тестирование (обязательно для каждого этапа)
- Chrome DevTools → Network → Offline
- Создать/редактировать/удалить задачу офлайн
- Восстановить сеть → проверить sync

### Автоматизированные проверки
- `npx tsc --noEmit` — после каждого изменения
- `npm run lint` — после каждого изменения
- Playwright E2e с offline mode (если настроен)

### Чеклист для каждого мутационного типа
- [ ] UI обновляется мгновенно (optimistic)
- [ ] Нет ошибок в консоли
- [ ] OfflineQueueError подавляется
- [ ] После восстановления сети данные синхронизируются
- [ ] Для create: серверный ID заменяет локальный
