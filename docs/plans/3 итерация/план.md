## Необходимые изменения (обоснование)

### Новые сущности

1. **Поля рекурренса в Task**
   - **Зачем:** Для хранения информации о том, как часто повторять задачу
   - **Минимальный набор:** только то, что нужно для базовых типов повторения
   - **Примеры recurrence_config:**
     ```json
     // Daily: каждые 2 дня
     {"type": "daily", "interval": 2}

     // Weekly: по понедельникам и средам
     {"type": "weekly", "interval": 1, "days": [1, 3]}

     // Monthly: 15-го числа
     {"type": "monthly", "interval": 1, "day_of_month": 15}

     // Monthly: второй четверг
     {"type": "monthly", "interval": 1, "week_of_month": 2, "day_of_week": 4}
     ```

2. **Поле timezone в User**
   - **Зачем:** Для корректного отображения времени напоминаний и дедлайнов в часовом поясе пользователя
   - **Значение по умолчанию:** "Europe/Moscow"
   - **Использование:** Все datetime хранятся в UTC, конвертируются в timezone пользователя при отправке уведомлений

3. **Таблица task_recurrences**
   - **Зачем:** История выполнений повторяющейся задачи (когда была сгенерирована, её статус)
   - **Почему не обойтись без:** Без этого нельзя восстановить историю и отследить пропущенные выполнения

4. **Таблица notifications**
   - **Зачем:** Хранение уведомлений для пользователей
   - **Почему не обойтись без:** Нужно для показа истории уведомлений и статуса прочитанности

5. **Notification**
   - **Зачем:** Объект уведомления для фронтенда
   - **Почему не обойтись без:** Браузер notifications — это ephemeral, нужно персистентное хранилище

### Новые зависимости

1. **APScheduler**
   - **Зачем:** Для периодического выполнения задач (генерация повторяющихся задач, отправка напоминаний)
   - **Почему не альтернативы:**
     - Celery — избыточно для in-process scheduled tasks
     - Вариант с crontab — не кроссплатформенно, сложно в docker
   - **Минимизация:** Используем только in-process scheduler через FastAPI lifespan

2. **sse-starlette**
   - **Зачем:** Для реализации Server-Sent Events
   - **Почему не WebSockets:** SSE проще для однонаправленного потока (server → client)
   - **Почему не polling:** SSE более эффективно (одно постоянное соединение vs частые запросы)

## Бизнес-правила и стратегии

### Генерация следующей задачи
- **Правило:** Генерировать СРАЗУ после выполнения текущей задачи
- **Обоснование:** Лучший UX — пользователь видит следующую задачу сразу, не дожидаясь due_date

### Обработка пропущенных выполнений
- **Правило:** Генерировать пропущенные задачи за последние 7 дней + все будущие задачи
- **Обоснование:** Баланс между восстановлением и перегрузкой пользователя
- **Реализация:** Метод `catch_up_missed_tasks(max_days=7)` в RecurrenceService

### Обработка DST (переход на летнее/зимнее время)
- **Стратегия:** Все datetime хранятся в UTC, конвертируются в user.timezone только для отображения и уведомлений
- **Правило:** 9:00 localtime всегда 9:00 localtime (включая DST коррекции)
- **Пример:** При переходе на летнее время, задача в 9:00 будет в 6:00 UTC (вместо 7:00 UTC)

### Предотвращение повторных напоминаний
- **Правило:** Проверять `last_reminder_sent_at` перед отправкой напоминания
- **Логика:** Не отправлять напоминание, если уже было отправлено за последние 24 часа для этой задачи

### Остановка рекурренции
- **Правило:** При stop-recurrence:
  - Оставить уже сгенерированные задачи (они уже существуют и могут быть выполнены)
  - Обновить исходную задачу: `recurrence_type = NULL`, `recurrence_config = NULL`
  - Создать notification пользователю о прекращении рекурренции

### Timezone handling
- **Правило:** Все datetime в БД хранятся в UTC
- **Конвертация:** При отправке уведомлений конвертировать в timezone пользователя
- **Validation:** Валидировать timezone через zoneinfo при создании/обновлении пользователя
- **Автоопределение:** Frontend определяет timezone при первом логине через `Intl.DateTimeFormat().resolvedOptions().timeZone`

### Offline sync — конфликт изменений
- **Стратегия:** Last write wins
- **Реализация:** Каждая мутация в offline queue имеет timestamp, выполняются по порядку
- **UX:** Показывать warning при sync, если есть potential conflict

### SSE error handling
- **Стратегия:** Exponential backoff с retry
- **Параметры:**
  - Начальная задержка: 1 секунда
  - Максимальная задержка: 30 секунд
  - Максимальное количество попыток: безлимитно (постоянное переподключение)
- **UI:** Показывать индикатор disconnected при потере соединения

### Notification TTL
- **Правило:** Автоматически удалять уведомления старше 30 дней
- **Реализация:** `job_cleanup_old_notifications()` выполняется раз в день
- **UX:** Показывать warning в UI, если уведомление скоро истечет (последние 3 дня)

## Integration между сервисами

### Backend integration
- **ReminderService → SSE:** ReminderService инжектирует SSEManager как зависимость для триггера событий
- **TaskService → ReminderService:** Вызывается в route handlers (API layer) после создания/обновления задачи
- **RecurrenceService → TaskService:** TaskService инжектирует RecurrenceService для генерации следующей задачи при mark_completed
- **Event flow:**
  1. User action → API endpoint
  2. API endpoint → TaskService/ReminderService
  3. Service → БД + SSE.trigger()
  4. SSE → Frontend via SSE connection

### Frontend state management
- **Store:** Zustand для глобального состояния
- **SSE integration:** При получении SSE события → emit event → Zustand store слушает и обновляет
- **Example:**
  ```typescript
  // Zustand store
  const useTaskStore = create((set) => ({
    tasks: [],
    updateTask: (task) => set((state) => ({
      tasks: state.tasks.map(t => t.id === task.id ? task : t)
    })),
    onSSEMessage: (message) => {
      if (message.type === 'task_updated') {
        useTaskStore.getState().updateTask(message.data)
      }
    }
  }))

  // SSE hook
  useSSE().subscribeToSync(userId, (message) => {
    useTaskStore.getState().onSSEMessage(message)
  })
  ```

### Timezone detection на фронтенде

// При первом логине или регистрации
const userTimezone = Intl.DateTimeFormat().resolvedOptions().timeZone
// Отправить на backend
await api.updateUser({ timezone: userTimezone })

## Известные ограничения и future improvements

### Текущие ограничения
1. **Single worker mode:** Scheduler должен работать в single worker mode для предотвращения race conditions
   - **Future:** Использовать distributed lock (Redis) для multi-worker support

2. **SSE scalability:** Ограниченное количество одновременных SSE соединений (~1000 на один сервер)
   - **Future:** Использовать Redis pub/sub или external service (Pusher, Ably)

3. **Offline sync conflicts:** Current strategy is last write wins
   - **Future:** Implement merge strategy or manual conflict resolution

4. **Recurrence config:** Поддерживаются только базовые типы (daily, weekly, monthly)
   - **Future:** Добавить поддержку complex patterns (cron-like, business days only)

5. **Notification channels:** Только in-app и browser notifications
   - **Future:** Email, SMS, push notifications (mobile)

### Future improvements (не в рамках v3)
1. **Snooze notifications:** Отложить напоминание на N минут/часов/дней
2. **Smart reminders:** ML-based предсказание лучшего времени для напоминания
3. **Recurrence templates:** Presets для common patterns (еженедельные встречи, ежемесячные отчеты)
4. **Notification digest:** Объединение нескольких уведомлений в одно (для снижения шума)
5. **Cross-device sync:** Синхронизация между мобильным и десктопом приложениями
