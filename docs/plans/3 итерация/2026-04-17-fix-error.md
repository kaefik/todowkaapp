## Детальный план исправления

### Задача 1: Исправить проблему с IndexedDB (ВЫСОКИЙ приоритет)

**Проблемы:**
- `DOMException: IDBDatabase.transaction: 'cache' is not a known object store name`
- `Encountered an error attempting to restore client cache from persisted location`

**Причина:** База данных 'todowka-query-cache' создана с устаревшей версией без object store 'cache'

**Решение:** Реализовать автоматическую миграцию с сохранением данных

**Шаги:**
1. Изменить версию базы данных с `1` на `2` в `frontend/src/lib/queryClient.ts`
2. Добавить обработчик `onupgradeneeded` для миграции данных:
   - Если есть старая база → перенести данные во временный объект store
   - Создать новый object store 'cache'
   - Перенести данные обратно
   - Удалить временный store
3. Добавить обработку ошибок миграции с автоматическим сбросом при критических ошибках

**Файлы:**
- `frontend/src/lib/queryClient.ts:31-42`

### Задача 2: Исправить авторизацию SSE (КРИТИЧЕСКИЙ приоритет)

**Проблемы:**
- `401 Unauthorized` для SSE endpoint
- `Firefox can't establish a connection to SSE`
- Бесконечный цикл переподключений

**Причины:**
1. Cookie `access_token` устанавливается с `secure=True` в dev режиме (HTTP не поддерживает secure cookies)
2. SSE подключается напрямую к `localhost:8000`, в обход Vite proxy
3. Нет механизма остановки переподключений при постоянной 401

**Решение:**

**Шаг 2.1: Условный secure для cookie**
1. Изменить `backend/app/security.py:58-67`
2. Добавить параметр `secure` на основе `settings.app_env`
3. В dev режиме: `secure=False`, в production: `secure=True`

**Шаг 2.2: Настроить SSE через Vite proxy**
1. Изменить `frontend/src/services/sseManager.ts:43-44`
2. В dev режиме использовать `/api/sse/notifications` (через proxy)
3. Убрать `withCredentials: true` в dev режиме (Vite proxy обрабатывает это)

**Шаг 2.3: Добавить лимит переподключений**
1. Изменить `frontend/src/services/sseManager.ts:84-92`
2. Добавить счетчик неудачных попыток
3. После N попыток (например, 5) остановить переподключения
4. Логировать финальную ошибку для отладки

**Файлы:**
- `backend/app/security.py:58-67`
- `backend/app/config.py:17` (добавить флаг `cookie_secure`)
- `frontend/src/services/sseManager.ts:43-44, 51, 84-92`
- `frontend/vite.config.ts:75-84` (проверить настройки proxy)

### Задача 3: Исправить авторизацию API (КРИТИЧЕСКИЙ приоритет)

**Проблема:** `401 Unauthorized` для всех API запросов

**Причина:** Токен отсутствует или недействителен (может быть следствием проблемы с cookie)

**Решение:**

**Шаг 3.1: Диагностика**
1. Проверить, устанавливается ли cookie при логине (DevTools → Application → Cookies)
2. Проверить, передается ли cookie в запросах (DevTools → Network)

**Шаг 3.2: Добавить fallback авторизации**
1. Если cookie не работает → использовать Authorization header (уже реализовано в `httpClient.ts:68-70`)
2. Убедиться, что токен из authStore правильно сохраняется

**Шаг 3.3: Добавить отладочные логи**
1. В `frontend/src/api/httpClient.ts:68-70` добавить лог токена
2. В `backend/app/dependencies.py:70-76` добавить лог полученного токена

**Файлы:**
- `frontend/src/stores/authStore.ts:46, 138, 168, 182, 214`
- `frontend/src/api/httpClient.ts:68-70`

## Проверка после исправлений

1. **Очистить IndexedDB** в браузере (DevTools → Application → Storage)
2. **Перелогиниться** в приложении
3. **Проверить Cookie** в DevTools → Application → Cookies:
   - Должен быть `access_token` cookie
   - Path: `/api/sse`
   - HttpOnly: true
   - Secure: false (в dev режиме)
4. **Проверить Network** вкладку:
   - API запросы должны возвращать 200 OK
   - SSE запрос должен возвращать 200 и держать соединение открытым
5. **Проверить Console**:
   - Не должно быть ошибок IndexedDB
   - Не должно быть ошибок SSE 401
   - Должны быть heartbeat сообщения от SSE
