## Обзор концепции

Skills — это специализированные наборы инструкций для конкретных типов задач. Они загружаются через инструмент `skill` и предоставляют детальные workflow, шаблоны и лучшие практики.

### Главное правило

Если есть хотя бы 1% вероятности, что скилл применим к текущей задаче — **его нужно вызвать**. Если после загрузки окажется, что он не подходит — можно не использовать.

### Приоритет инструкций

1. **Пользовательские инструкции** (AGENTS.md, CLAUDE.md, прямые запросы) — высший приоритет
2. **Superpowers skills** — переопределяют поведение по умолчанию при конфликте
3. **Системный промпт** — низший приоритет

### Типы скиллов

| Тип | Описание | Примеры |
|------|----------|---------|
| **Жёсткие (Rigid)** | Следовать точно, не отступать от дисциплины | TDD, debugging |
| **Гибкие (Flexible)** | Адаптировать принципы к контексту | patterns, brainstorming |

---

## Порядок применения

Когда несколько скиллов могут подойти одновременно, использовать в таком порядке:

1. **Процессные скиллы** (brainstorming, debugging) — определяют КАК подойти к задаче
2. **Скиллы реализации** (design-to-plan, task-executor) — направляют выполнение

Пример: «Давай создадим X» → сначала brainstorming, затем скиллы реализации.

## Процесс разработки

### 1. brainstorming

**Файл скилла:** `skills/brainstorming/SKILL.md`

**Когда использовать:**
- Перед любой творческой работой
- Создание новых фич, компонентов
- Добавление функциональности
- Изменение поведения существующего кода
- Любая задача, где нужно придумать решение, а не просто выполнить известные шаги

**Зачем:** Исследует намерения пользователя, требования и дизайн до начала реализации. Предотвращает поспешное написание кода без понимания задачи.

**Что делает:**
- Задаёт уточняющие вопросы о требованиях
- Помогает определить масштаб и границы задачи
- Выявляет неочевидные требования и крайние случаи
- Формирует общее понимание проблемы до начала работы

**Когда НЕ нужен:**
- Багфикс с понятной причиной
- Рефакторинг с чётким планом
- Добавление тестов к существующему коду
- Ответы на информационные вопросы

---

### 2. brainstorming-skill

**Файл скилла:** `skills/brainstorming-skill/SKILL.md`

**Когда использовать:**
- Генерация идей и креативных решений
- Систематический мозговой штурм
- Планирование контента
- Продуктовые фичи — какой функционал добавить
- Маркетинговые кампании
- Стратегическое планирование
- Креативное письмо
- Любая задача, требующая структурированной генерации идей

**Зачем:** Предоставляет 30+ исследовательски-валидированных шаблонов промптов по 14 категориям с точными шаблонами, метриками успеха и доменно-специфическими приложениями.

**Отличие от brainstorming:**
- `brainstorming` — базовый скилл для понимания задачи перед реализацией
- `brainstorming-skill` — продвинутый инструментарий для генерации идей с шаблонами

---

### 3. writing-plans

**Файл скилла:** `skills/writing-plans/SKILL.md`

**Когда использовать:**
- Есть спецификация или требования для многошаговой задачи
- Перед тем как писать код
- Нужно спланировать последовательность шагов

**Зачем:** Создание структурированного плана реализации до начала кодирования. Помогает избежать хаотичной разработки и пропущенных шагов.

**Что делает:**
- Разбивает задачу на последовательные шаги
- Определяет зависимости между шагами
- Формирует чёткий план действий

---

### 4. design-to-plan

**Файл скилла:** `skills/design-to-plan/SKILL.md`

**Когда использовать:**
- Есть дизайн-документ (API дизайн, архитектура приложения, системная спецификация, PRD, техническая спецификация, ERD или любой планирующий документ)
- Нужно разбить документ на задачи, подзадачи, roadmap, шаги реализации, sprint plan или execution checklist

**Триггерные фразы пользователя:**
- «разбить на задачи»
- «составить план реализации»
- «пошаговый план»
- «декомпозировать документ»
- «break into tasks»
- «create implementation plan»
- «how do I implement this design»

**Зачем:** Декомпозиция документа в пошаговый план реализации, где каждый шаг атомарный — достаточно мал для одного промпта LLM или одной задачи разработчика (максимум 1-4 часа).

**Что делает:**
- Анализирует дизайн-документ
- Выделяет атомарные задачи
- Определяет порядок выполнения
- Создаёт структурированный план

---

### 5. plan-critic

**Файл скилла:** `skills/plan-critic/SKILL.md`

**Когда использовать:**
- Критический анализ плана, дизайна или спецификации перед реализацией
- Перед тем как начать писать код по плану

**Триггерные фразы пользователя:**
- «find holes in this plan»
- «challenge my assumptions»
- «what am I missing»
- «critique this design»
- «провести критический разбор»
- «найти слабые места»
- «что я упустил»
- «проверить план»
- «оценить дизайн»

**Зачем:** Поиск слабых мест, пропущенных деталей, ошибочных предположений в плане ДО начала реализации. Исправление ошибок на этапе планирования на порядки дешевле, чем после написания кода.

**Когда запускать:** 1-2 раза после brainstorming и перед design-to-plan.

**Результат:** Структурированный документ с критикой и вердикт:
- **APPROVED** — план готов к реализации
- **CONDITIONAL** — можно начинать с оговорками
- **NEEDS REVISION** — план требует переработки

---

### 6. executing-plans

**Файл скилла:** `skills/executing-plans/SKILL.md`

**Когда использовать:**
- Есть готовый план реализации
- Нужно выполнить его в отдельной сессии с контрольными точками

**Зачем:** Управляемое выполнение плана с промежуточными проверками (review checkpoints). Позволяет отслеживать прогресс и корректировать курс.

**Что делает:**
- Выполняет план пошагово
- Останавливается на контрольных точках для проверки
- Позволяет вносить корректировки

---

### 7. task-executor

**Файл скилла:** `skills/task-executor/SKILL.md`

**Когда использовать:**
- Выполнение задач из Kanban-структуры в папке `tasks/`
- Работа с файловыми задачами

**Триггерные фразы:**
- «следующая задача», «запусти задачу», «что дальше»
- «сгенерируй промпт», «начни разработку»
- «выполни задачу», «задача готова», «отметь выполненной»
- «покажи прогресс», «start next task», «run task»
- «execute task», «mark done», «show metrics»

**Зачем:** Управление задачами через Kanban-структуру директорий:
- `tasks/backlog/` — задачи в очереди
- `tasks/ready/` — задачи, готовые к выполнению
- `tasks/in_progress/` — задачи в работе
- `tasks/done/` — завершённые задачи
- `tasks/blocked/` — заблокированные задачи

Каждая задача — отдельный `.md` файл. Прогресс отслеживается через `tasks/_progress.json`.

---

### 8. subagent-driven-development

**Файл скилла:** `skills/subagent-driven-development/SKILL.md`

**Когда использовать:**
- 2+ независимых задач
- Задачи можно выполнять параллельно
- Задачи не имеют общих данных или последовательных зависимостей

**Зачем:** Параллельное выполнение задач через субагентов. Ускоряет разработку за счёт одновременной работы над независимыми частями проекта.

---

### 9. dispatching-parallel-agents

**Файл скилла:** `skills/dispatching-parallel-agents/SKILL.md`

**Когда использовать:**
- Несколько независимых задач без общих данных
- Нет последовательных зависимостей между задачами
- Нужно максимизировать скорость выполнения

**Зачем:** Диспетчеризация параллельных агентов для одновременной работы. Похож на subagent-driven-development, но фокусируется на диспетчеризации — распределении задач по агентам.

## Качество и проверка

### 10. test-driven-development (TDD)

**Файл скилла:** `skills/test-driven-development/SKILL.md`

**Когда использовать:**
- Перед реализацией любой фичи
- Перед исправлением любого бага
- Всегда до написания кода реализации

**Зачем:** Сначала тесты, потом реализация. Обеспечивает:
- Покрытие кода тестами с самого начала
- Чёткое понимание ожидаемого поведения
- Безопасный рефакторинг
- Документирование поведения кода через тесты

**Тип скилла:** Жёсткий (Rigid) — следовать точно, не отступать.

---

### 11. systematic-debugging

**Файл скилла:** `skills/systematic-debugging/SKILL.md`

**Когда использовать:**
- Столкнулись с багом
- Тест падает с непонятной ошибкой
- Неожиданное поведение программы
- Проблемы с производительностью
- Любая ситуация, где «что-то работает не так»

**Зачем:** Систематический подход к отладке. Предотвращает «слепое» исправление — угадывание и тыканье пальцем в небо.

**Что делает:**
- Структурирует процесс отладки
- Помогает локализовать проблему
- Предлагает методичный подход к поиску корневой причины
- Формулирует гипотезы и проверяет их

**Тип скилла:** Жёсткий (Rigid) — следовать точно.

---

### 12. verification-before-completion

**Файл скилла:** `skills/verification-before-completion/SKILL.md`

**Когда использовать:**
- Перед тем как заявить, что работа завершена
- Перед тем как сказать «исправлено» или «тесты проходят»
- Перед коммитом или созданием PR

**Зачем:** Обязательная проверка через команды верификации (lint, typecheck, тесты). Принцип: **доказательства вместо утверждений**. Никаких «должно работать» — только проверенный результат.

**Что делает:**
- Запускает соответствующие команды проверки
- Анализирует результат
- Подтверждает или опровергает успешность

---

### 13. qa-checklist

**Файл скилла:** `skills/qa-checklist/SKILL.md`

**Когда использовать:**
- После завершения реализации
- Нужно проверить, что все фичи реализованы
- Перед релизом

**Триггерные фразы:**
- «создай чеклист проверки»
- «сгенерируй QA чеклист»
- «проверь реализацию»
- «create verification checklist»
- «what should I test»
- «did we implement everything»

**Зачем:** Генерация комплексного чеклиста для проверки качества. Принимает дизайн-документ и план реализации как входные данные.

**Результат:** Двойной чеклист:
- **Ручное тестирование** — шаги для ручной проверки
- **Автоматические тесты** — спецификации для автотестов

**Когда запускать:** После завершения всех задач через task-executor.

---

### 14. requesting-code-review

**Файл скилла:** `skills/requesting-code-review/SKILL.md`

**Когда использовать:**
- Завершена задача
- Реализована крупная фича
- Перед мёрджем в основную ветку

**Зачем:** Структурированный запрос code review. Помогает получить качественный обзор кода, а не поверхностные комментарии.

---

### 15. receiving-code-review

**Файл скилла:** `skills/receiving-code-review/SKILL.md`

**Когда использовать:**
- Получили feedback от code review
- Перед реализацией предложений из ревью
- Особенно если фидбек кажется неясным или технически сомнительным

**Зачем:** Технически строгая проверка фидбека. Не слепое согласие и реализация любых предложений, а:
- Верификация корректности замечаний
- Проверка, что предложение действительно улучшит код
- Обоснованное согласие или несогласие

## Управление проектом

### 16. project-onboarding

**Файл скилла:** `skills/project-onboarding/SKILL.md`

**Когда использовать:**
- Нужно применить workflow к существующему проекту
- Реверс-инжиниринг кодовой базы для планирования
- Документирование существующего проекта

**Триггерные фразы:**
- «внедри workflow в проект»
- «подключи workflow к существующему проекту»
- «onboard existing project»
- «retrofit this codebase»
- «document this project for planning»

**Режимы работы:**

| Режим | Описание | Когда |
|-------|----------|-------|
| **MODE A — Feature** | Документирует только то, что нужно для добавления конкретной фичи | Нужно добавить одну фичу в существующий проект |
| **MODE B — Full** | Полный реверс-инжиниринг всего проекта | Комплексное планирование развития проекта |

**Результат:** Документация проекта, готовая для подачи в pipeline: `plan-critic → design-to-plan → task-executor`.

### 17. project-portability

**Файл скилла:** `skills/project-portability/SKILL.md`

**Когда использовать:**
- Нужно портировать проект на другую платформу

**Что делает:**
- Анализирует архитектуру проекта
- Выделяет платформенно-независимую логику
- Определяет платформенно-зависимые компоненты
- Создаёт задачи для портирования

## Типичный pipeline

Полный рабочий процесс от идеи до готовой фичи:

```
┌─────────────────────────────────────────────────────────────────────┐
│  1. BRAINSTORMING                                                    │
│     Понимаем задачу, собираем требования                            │
│     ↓                                                                │
│  2. PLAN-CRITIC                                                      │
│     Критикуем план, ищем дыры (1-2 итерации)                        │
│     ↓                                                                │
│  3. DESIGN-TO-PLAN                                                   │
│     Декомпозируем на атомарные задачи (1-4 часа каждая)             │
│     ↓                                                                │
│  4. TASK-EXECUTOR / SUBAGENT-DRIVEN-DEVELOPMENT                      │
│     Выполняем задачи (с TDD для каждой)                             │
│     ↓                                                                │
│  5. VERIFICATION-BEFORE-COMPLETION                                   │
│     Проверяем: линтер, типы, тесты                                   │
│     ↓                                                                │
│  6. QA-CHECKLIST                                                     │
│     Финальная проверка: ручное + автоматическое тестирование        │
│     ↓                                                                │
│  7. FINISHING-A-DEVELOPMENT-BRANCH                                   │
│     Merge / PR / Cleanup                                             │
└─────────────────────────────────────────────────────────────────────┘
```

### Альтернативные сценарии

**Багфикс:**
```
systematic-debugging → test-driven-development → verification-before-completion
```

**Портирование проекта:**
```
project-onboarding (MODE B) → project-portability → design-to-plan → task-executor
```

**Code review:**
```
requesting-code-review → (получили фидбек) → receiving-code-review → реализация → verification-before-completion
```

**Потеря контекста:**
```
session-snapshot (сохранить) → ...новая сессия... → session-snapshot (восстановить)
```

## Антипаттерны

Мысли, которые означают СТОП — вы рационализируете:

| Мысль | Реальность |
|-------|-----------|
| «Это просто простой вопрос» | Вопросы — это задачи. Проверьте скиллы. |
| «Мне нужен больше контекста» | Проверка скиллов ДО уточняющих вопросов. |
| «Давай сначала изучу код» | Скиллы говорят КАК изучать. Проверьте сначала. |
| «Я могу быстро проверить git/файлы» | Файлы не имеют контекста разговора. |
| «Это не требует формального скилла» | Если скилл существует — используйте его. |
| «Я помню этот скилл» | Скиллы обновляются. Читайте текущую версию. |
| «Это не считается за задачу» | Действие = задача. Проверьте скиллы. |
| «Скилл — это избыточность» | Простые вещи становятся сложными. Скиллы предотвращают это. |
| «Сначала сделаю вот это» | Проверьте ПЕРЕД тем как что-либо делать. |
| «Это кажется продуктивным» | Недисциплинированные действия тратят время. |
