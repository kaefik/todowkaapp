# Анализ плана sync engine для однопользовательского offline-first PWA

## 🔴 Противоречия и проблемные места

### 1. Entity-level LWW — слишком грубая гранулярность

В плане написано: *"UPDATE → entity-level LWW: compare updated_at, newer wins"*

**Проблема:** Если на устройстве A ты изменил название задачи, а на устройстве B — её приоритет, entity-level LWW выбросит одно из изменений целиком, потому что сравнивается один `updated_at` на всю сущность. Для однопользовательского приложения это особенно обидно — ты сам себе теряешь данные.

**Почему это важно:** Однопользовательское приложение означает, что конфликты возникают не от разных людей, а от тебя самого на разных устройствах. Ты ожидаешь, что все твои изменения сохранятся. Entity-level LWW нарушает это ожидание.

**Рекомендация:** Либо перейти на **field-level LWW** (каждое поле имеет свой `updated_at`), либо хотя бы делать **field-level merge**: при конфликте сравнивать поля и сливать неконфликтующие изменения, показывая конфликт только для полей, изменённых на обоих устройствах.

---

### 2. "Delete always wins over concurrent updates" — опасная семантика

**Проблема:** Представь сценарий: на устройстве A ты удалил задачу, а на устройстве B (офлайн) ты добавил к ней подробное описание, подзадачи, вложил усилия. При синке удаление победит — вся работа пропадёт молча.

**Почему это плохо для однопользовательского:** Ты один. Если ты редактировал задачу на одном устройстве, значит ты передумал её удалять или забыл, что удалил на другом. Молчаливая потеря данных — это худший UX.

**Рекомендация:** При конфликте "delete vs update" — не удалять автоматически, а показывать конфликт пользователю: "Вы удалили эту задачу на другом устройстве, но здесь внесли изменения. Что делать?" Или хотя бы soft-delete с возможностью восстановления + уведомление.

---

### 3. Конфликты "show in UI (Phase 5+)" — а данные теряются уже в Phase 4

В Phase 4 работает LWW и "delete always wins", но UI для конфликтов запланирован только на Phase 5+. Это значит, что на Phase 4 **данные будут молча перезаписываться или удаляться без возможности увидеть, что произошло**.

**Рекомендация:** Как минимум логировать конфликты в IndexedDB с Phase 4 и дать хотя бы простейший список "что было перезаписано при последней синхронизации".

---

### 4. Зависимость от серверных часов в LWW

*"compare updated_at, newer wins"* — но `updated_at` на клиенте и сервере могут расходиться. Часы на телефоне могут быть выставлены неправильно (особенно при путешествиях, ручной настройке).

**Для однопользовательского приложения** это реальный риск: если на одном устройстве часы отстают на час, все его изменения будут проигрывать более "свежим" изменениям с другого устройства.

**В плане нет** никакого упоминания о clock skew handling (в тестах есть "clock skew scenarios", но нет описания, как это решается в дизайне).

**Рекомендация:** Использовать **логические часы** (например, Lamport timestamp или простой монотонный счётчик на устройство) вместо wall-clock `updated_at`, либо нормализовать клиентское время по серверному ответу.

## 🟡 Узкие места

### 5. Dependency-ordered processing — хрупкость при частичных сбоях

```
Server processes pending_ops IN DEPENDENCY ORDER:
1. areas → 2. contexts → 3. tags → 4. projects → 5. tasks → 6. subtasks
```

**Что, если** создание project (шаг 4) падает, а в батче есть tasks (шаг 5), ссылающиеся на этот project? План не описывает, что происходит:
- Откатываются все операции?
- Пропускаются зависимые операции?
- Частично применяется?

**Рекомендация:** Явно описать поведение: применять в транзакции (всё или ничего) или описать стратегию partial-apply с возвратом ошибок для непримёненных операций.

---

### 6. "Max 100 pending_ops per request, max 10KB per op.data" + "Large pending ops (>100) → batch into chunks of 50"

**Противоречие в числах:** Лимит 100 ops на запрос, но батчинг по 50. Это непротиворечиво, но неясно зачем батчить по 50, если лимит 100. Вероятно 50 — это просто "с запасом", но это нигде не объяснено.

**Реальная проблема:** Если пользователь был офлайн неделю и накопил 500 операций, нужно 10 последовательных синков. Каждый синк меняет `last_sync_timestamp`, и следующий батч может получить server_changes, которые конфликтуют с ещё неотправленными локальными операциями. **Порядок батчей и обновление timestamp между ними не описаны.**

**Рекомендация:** Описать протокол мультибатчевого синка: не обновлять `last_sync_timestamp` до отправки всех батчей, или отправлять все батчи с одним и тем же timestamp.

---

### 7. SSE push notification (Phase 5) как sync trigger

SSE требует постоянного открытого HTTP-соединения. На мобильных устройствах (особенно iOS Safari) это **ненадёжно** — браузер убивает фоновые соединения при сворачивании. А в standalone PWA режиме на iOS ещё жёстче.

**Рекомендация:** Для однопользовательского приложения SSE — это overreach. Periodic polling + visibilitychange покрывают 95% сценариев. Если очень нужен push — рассмотреть Web Push API (хотя бы на Android/Chrome).

---

## 🟠 Оверинжиниринг

### 8. Background Sync API — сложность ради маргинального улучшения

Background Sync работает **только в Chrome** (план это признаёт). Safari/iOS не поддерживает. Для однопользовательского приложения, где синк не критичен по времени, `online` + `visibilitychange` + periodic (5 мин) + manual — **уже более чем достаточно**.

**Почему оверинжиниринг:** Это добавляет код в Service Worker, усложняет тестирование (нужно мокать SW sync events), а выигрыш — синк произойдёт на 0-5 минут быстрее, только в Chrome.

**Рекомендация:** Убрать из Phase 4, оставить как "nice to have" для будущего.

---

### 9. `sync_in_progress` stale flag recovery — решение несуществующей проблемы при правильном подходе

Весь блок `recoverStaleSyncFlag()` существует потому, что `sync_in_progress` хранится как **персистентное состояние в IndexedDB**. 

**Вопрос:** Зачем sync mutex вообще хранить в IndexedDB? Это **рантайм-состояние**. Простая переменная `let syncInProgress = false` в памяти:
- Автоматически сбрасывается при краше/перезагрузке
- Не требует recovery-логики
- Не требует `sync_started_at`
- Работает как mutex в рамках одной вкладки

**Единственный случай, когда нужен персистентный mutex** — если несколько вкладок могут синкать одновременно. Но это решается через `BroadcastChannel` или `navigator.locks` API, а не через IndexedDB с 5-минутным таймаутом.

**Рекомендация:** Заменить на in-memory флаг + `navigator.locks` API для кросс-табовой координации. Удалить весь `recoverStaleSyncFlag`.

---

### 10. Conflicts array в ответе — на Phase 4 бесполезен

Сервер возвращает `conflicts: [{ entity_type, server_data, local_data, reason }]`, но:
- LWW автоматически выбирает победителя — конфликт уже разрешён
- UI для конфликтов — Phase 5+
- На Phase 4 конфликты некуда деть

**Рекомендация:** На Phase 4 достаточно просто возвращать `applied` и `server_changes`. Конфликты добавить в API, когда появится UI для них.

---

### 11. Шесть (!) sync triggers — избыточно для одного пользователя

| Trigger | Реально нужен? |
|---------|---------------|
| `online` event | ✅ Да, основной |
| Background Sync | ❌ Только Chrome, маргинальный выигрыш |
| `visibilitychange` | ✅ Да, полезен |
| Periodic 5 мин | ✅ Да, как fallback |
| Manual button | ✅ Да, для спокойствия пользователя |
| SSE push | ⚠️ Сложно, ненадёжно на iOS, для одного юзера избыточно |

4 из 6 — достаточно. Два остальных добавляют сложность без пропорциональной пользы.

## 📋 Резюме

| Категория | Проблема | Серьёзность |
|-----------|----------|-------------|
| Противоречие | Entity-level LWW теряет данные при параллельном редактировании разных полей | 🔴 Высокая |
| Противоречие | "Delete always wins" молча удаляет отредактированные данные | 🔴 Высокая |
| Противоречие | Конфликты показываются в Phase 5, но данные теряются в Phase 4 | 🔴 Высокая |
| Узкое место | Нет обработки clock skew в дизайне | 🟡 Средняя |
| Узкое место | Поведение при частичном сбое dependency chain не описано | 🟡 Средняя |
| Узкое место | Мультибатчевый синк не описан | 🟡 Средняя |
| Оверинжиниринг | Background Sync API | 🟠 Низкая |
| Оверинжиниринг | Персистентный sync mutex + recovery | 🟠 Низкая |
| Оверинжиниринг | SSE для одного пользователя | 🟠 Низкая |
| Оверинжиниринг | Conflicts в API до появления UI | 🟠 Низкая |

**Главный вывод:** Для однопользовательского приложения ядро конфликт-резолюции (LWW + delete wins) — слишком агрессивное. Ты единственный пользователь, и каждое потерянное изменение — это **твоё** изменение. Стоит инвестировать в field-level merge и более осторожную обработку удалений, а сэкономить сложность на уменьшении количества sync triggers и упрощении sync mutex.
